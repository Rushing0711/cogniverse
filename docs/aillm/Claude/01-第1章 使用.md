# 使用

## 1 模式切换

Claude Code 有 5 种权限模式，理解它们的区别是用好 CC 的基础。

### 1.1 五种模式速查

| 模式 | 启动参数 | 行为 |
|------|----------|------|
| **default**（默认） | `claude` | 每次文件写入、Bash 命令都弹确认框 |
| **acceptEdits** | `claude --permission-mode acceptEdits` | 自动接受文件编辑 + 工作目录内常见命令，其他操作仍确认 |
| **auto** | `claude --permission-mode auto` | 后台分类器自动判断是否安全，只拦截高风险操作 |
| **plan** | `claude --permission-mode plan` | 只读模式：只能读文件、搜索、分析，禁止任何编辑和执行 |
| **bypassPermissions** | `claude --dangerously-skip-permissions` | 跳过所有检查，Claude 完全自主 |

`--dangerously-skip-permissions` 有两个变体，经常被混淆：

```bash
# 变体1：启动时先正常确认，但允许会话中后续开启全权限
claude --allow-dangerously-skip-permissions
# 会话中使用 /permissions 或按 Shift+Tab 三次切换到 bypassPermissions

# 变体2：直接以全权限启动，任何时候都跳过确认
claude --dangerously-skip-permissions
```

**区别**：`--allow-` 是"允许但不默认开启"，进入会话后仍需要手动切换。`--dangerously-` 是"直接开启"，没有任何确认。本地开发用 `--allow-` 更安全，只在 Docker/CI 中用 `--dangerously-`。

![自动编辑模式](images/image-20260422130847868.png)

### 1.2 会话内快捷键切换

- 按 `Shift+Tab` **一次**：切换到 **acceptEdits** 模式（自动编辑）
- 按 `Shift+Tab` **两次**：切换到 **plan** 模式（先规划，如上图）
- 按 `Shift+Tab` **三次**：切换到 **bypassPermissions** 模式（需 `--allow-dangerously-skip-permissions` 启动）
- 再按一次：切回默认模式

快捷键在 default → acceptEdits → plan → bypassPermissions 之间轮换（bypassPermissions 仅当启动时加了 `--allow-dangerously-skip-permissions` 才会出现在轮换中）。auto 模式通过启动参数或 `/permissions` 命令设置。

### 1.3 什么场景用什么模式

```bash
# 新项目、不确定 Claude 会干什么 — 默认模式，每步确认
claude

# 日常开发、项目已用 git 管理 — 自动编辑，不怕改错
claude --permission-mode acceptEdits

# 需要深度思考、不想被确认框打断 — auto 模式，只拦高风险操作
claude --permission-mode auto

# 只想让 Claude 读代码出方案，别动手 — plan 模式
claude --permission-mode plan

# 日常开发想保留"全权限"选项 — 先正常启动，需要时再切换
claude --allow-dangerously-skip-permissions

# Docker/CI 环境，隔离好了随便跑 — 全权限直接启动
claude --dangerously-skip-permissions
```

### 1.4 实战演示

同一需求 `创建一个 React todolist 应用`：

- **default**：创建 App.tsx 弹确认 → 创建 TodoList.tsx 弹确认 → 安装依赖弹确认... 全程大约被问 8-10 次
- **acceptEdits**：一路创建文件不打断，但 `npm install` 需要确认一次；偶尔改错方向需要中途纠正
- **plan**：先读项目结构，输出组件树和状态管理方案。你觉得方案 OK 后切到 acceptEdits 执行

![Plan模式规划示例](images/image-20260422215016770.png)

![Plan模式输出结果](images/image-20260422215445629.png)
- **auto**：所有操作自动放行，包括 npm install。中途改错文件，但能快速纠正。效率最高但需要留意 git diff
- **bypassPermissions**：Docker 里跑全流程，从创建项目到运行测试一气呵成

**经验法则**：新项目用 plan 先规划 → acceptEdits 实现 → git 兜底，出错就 `git reset --hard`。

### 1.5 自定义允许命令

除了选择模式，还可以精细控制哪些操作免确认。在 `~/.claude/settings.json`（全局）或 `.claude/settings.local.json`（项目级，不提交 git）中配置。以下是一个兼顾安全与便利的生产级示例：

```json
{
  "permissions": {
    "allow": [
      "Edit",
      "WebSearch",
      "WebFetch(domain:skills.sh)",
      "Bash(npx skills *)",
      "Read(~/.claude/**)",
      "Bash(pnpm *)",
      "Bash(git *)",
      "Bash(ls *)",
      "Bash(node !-e !--eval *)",
      "Bash(find !-exec !-delete *)"
    ],
    "deny": [
      "Bash(git push --force)",
      "Bash(git push -f)",
      "Bash(git reset --hard)",
      "Bash(git clean)",
      "Bash(git branch -D)"
    ]
  }
}
```

**规则格式说明**：

| 规则 | 效果 |
|------|------|
| `"Edit"` | 允许所有文件编辑 |
| `"WebFetch(domain:skills.sh)"` | 限定域名抓取，`*` 通配 |
| `"Read(~/.claude/**)"` | 允许读取指定目录下所有文件 |
| `"Bash(pnpm *)"` | 允许所有 pnpm 命令，`*` 通配符 |
| `"Bash(git *)"` | 允许所有 git 命令（结合 deny 排除危险操作） |
| `"Bash(node !-e !--eval *)"` | 允许 node 但排除 `-e`/`--eval`，`!` 否定匹配 |
| `"Bash(find !-exec !-delete *)"` | 允许 find 但排除 `-exec`/`-delete` 危险参数 |

**`!` 否定模式**：在 allow 规则中使用 `!` 前缀，表示"允许但排除"。`"Bash(node !-e *)"` 等于"允许 node 命令，但不允许含 `-e` 参数的 node 命令"。

**`deny` 精确拦截**：deny 优先级高于 allow，适合白名单较宽时精准排除风险项：

```json
"deny": [
  "Bash(git push --force)",  // 阻止强制推送
  "Bash(git reset --hard)",  // 阻止硬重置
  "Bash(git clean)",         // 阻止清理未跟踪文件
  "Bash(git branch -D)"      // 阻止强制删除分支
]
```

**项目级 vs 用户级**：
- `~/.claude/settings.json` — 所有项目生效（你当前的配置已经有 `skipAutoPermissionPrompt`）
- `.claude/settings.json` — 仅当前项目，可检入 git 共享
- `.claude/settings.local.json` — 仅当前项目，不提交

**更细粒度控制**：用 `!` 否定模式在 allow 内部排除子命令（如 `node !-e *`），用 `deny` 数组明确禁止特定操作（deny 优先级高于 allow），或用 `/permissions` 在会话中动态调整。三条防线：allow 放行 → `!` 排除 → deny 拦截。

**不想写配置文件？启动参数直接控制**：

```bash
# 启动时指定允许的工具（逗号分隔），本次会话有效，不写入配置文件
claude --allowedTools "Edit, Bash(npm run *), Bash(git commit:*), WebFetch"

# 也可以组合：允许一类操作但禁止其中某项
claude --allowedTools "Edit" --disallowedTools "Edit(src/core/*)"

# 适合场景：临时给一个会话高权限但不想改配置
claude --allowedTools "Edit, Bash(*)" --permission-mode acceptEdits
```

`--allowedTools` 和 `--disallowedTools` 仅对本次会话生效，退出即忘。适合临时需要放宽/收紧权限的场景。

## 2 核心命令速查

### 2.1 Slash 命令（会话内使用）

| 命令 | 用途 |
|------|------|
| `/help` | 列出所有命令和快捷键 |
| `/clear` | 重置上下文，开始新对话 |
| `/compact` | 压缩对话历史，保留关键信息，释放 50-70% token |
| `/context` | 查看当前上下文使用情况（token 用量、已加载文件） |
| `/cost` | 显示当前会话 token 消耗和预估费用 |
| `/init` | 初始化项目，自动生成 CLAUDE.md |
| `/memory` | 编辑 CLAUDE.md 记忆文件 |
| `/rewind` | 回退到上一个检查点 |
| `/resume` | 恢复之前的会话 |
| `/diff` | 交互式查看未提交的代码变更 |
| `/plan` | 切换到 Plan 模式 |
| `/agents` | 管理子代理（创建、编辑、监控） |
| `/mcp` | 管理 MCP 服务器连接 |
| `/permissions` | 查看和调整权限规则 |
| `/doctor` | 诊断安装健康状态 |
| `/review` | 本地审查 PR |
| `/simplify` | 审查最近修改并修复质量/效率问题 |

### 2.2 CLI 启动参数

```bash
claude                           # 进入交互模式
claude -c                        # 继续上次会话
claude -r                        # 选择历史会话恢复
claude -p "分析代码"              # 无头模式，执行一次退出
claude -p "查询" --output-format json  # 结构化输出
cat log.txt | claude -p "分析日志"     # 管道输入
claude --model sonnet             # 指定模型
claude --resume <session-id>      # 恢复指定会话
claude --worktree feature-x       # 在独立 git worktree 中启动
```

**实战演示**：一次典型会话中的命令链 — `claude -c` 恢复上次进度 → `/context` 发现 token 用了 70% → `/compact` 压缩后继续 → 完成任务 `/diff` 查看变更 → `/clear` 为新任务让位。

### 2.3 在提示中直接执行 Shell 命令

在 Claude Code 交互会话中，用 `!` 前缀可以直接执行 shell 命令，无需离开对话：

```bash
# 在 CC 会话中直接输入
! ls                          # 列出当前目录文件
! git status                  # 快速查看仓库状态
! npm test                    # 运行测试
! cat src/config.ts           # 快速查看文件内容
! grep -r "TODO" src/         # 搜索项目中的 TODO
```

`!` 命令的输出会直接显示在对话中，不会交给 Claude 处理。适合快速查看状态、运行测试等无需 AI 介入的操作。需要 Claude 分析结果时，用正常提示："运行 npm test 并分析失败的用例"。

## 3 高效 Prompt 技巧

### 3.1 差 vs 好

| 差 | 好 |
|----|-----|
| 为 foo.py 添加测试 | 为 foo.py 编写新的测试用例，覆盖用户已注销的边缘情况。避免使用 mock |
| 为什么 ExecutionFactory 有奇怪的 API？ | 查看 ExecutionFactory 的 git 历史，总结它的 API 是怎么形成的 |
| 添加日历小部件 | 先查看首页现有小部件的实现方式，理解代码和接口的分离模式。参考 HotDogWidget.php，然后实现一个日历小部件，支持月份选择和翻页选年份。只用项目中已引入的库 |

具体 = 预期结果 + 约束条件 + 参考样本。

### 3.2 CC 特有的输入技巧

```bash
# 1. Tab 补全引用文件 — 在提示中按 Tab 快速引用项目文件
"重构 src/utils/auth.ts 中的 validateToken 函数"

# 2. 粘贴 URL — Claude 会抓取页面内容
"参考 https://docs.foo.com/api 实现这个接口"

# 3. 拖入/粘贴图像 — 设计稿、截图、架构图均可
"按这个设计稿实现登录页面"（直接粘贴截图）

# 4. 管道输入 — 适合大批量数据
cat error.log | claude -p "找出所有超时错误并分析原因"
```

### 3.3 修正方向

不要等 Claude 跑偏了再重来。四个工具帮你中途纠偏：

- **按 Esc** — 随时中断 Claude（思考中、工具调用中、文件编辑中）
- **按 Esc 两次** — 跳回历史对话节点，编辑之前的提示重来
- **要求撤销** — "回滚到上次的代码"
- **要求先出方案** — "先制定计划，我确认后再编码"

### 3.4 "Think" 关键词

在提示中加入这些关键词，给 Claude 分配更多思考预算。级别越高，Claude 花更多时间评估替代方案。

| 关键词 | 适用场景 |
|--------|----------|
| `think` | 一般问题，需要多考虑几步 |
| `think hard` | 跨多个文件的改动，需要权衡方案 |
| `think harder` | 架构设计、复杂重构 |
| `ultrathink` | 系统级方案，需要深度评估所有替代路径 |

**使用示例**：

```text
# think — 一般代码问题
"think，分析 src/utils/cache.ts 的潜在竞态条件，给出修复方案"

# think hard — 多文件重构
"think hard，把用户认证从 Session 迁移到 JWT，要考虑所有受影响的路由和中间件"

# think harder — 架构设计
"think harder，设计一个消息队列系统的重试策略，包含指数退避、死信队列、幂等性方案"

# ultrathink — 系统性方案
"ultrathink，评审整个数据访问层的安全性，识别所有可能的 SQL 注入点、权限越权和数据泄露风险"
```

**实际效果**：测试同一个问题"设计 API 限流方案"，不加 think → 给一个简单的令牌桶实现；加 `think hard` → 输出令牌桶 + 滑动窗口 + 漏桶三种方案的利弊对比，附带 Redis + Lua 的实现思路。

**实战演示**：一个反面示例 `添加用户登录功能`，Claude 可能写一堆你不需要的代码；正面示例 `参考 src/auth/oauth.ts 的实现模式，为 admin 模块添加 JWT 登录，包含 token 刷新和过期处理，在 CLAUDE.md 查看项目规范`，Claude 一次到位。

## 4 CLAUDE.md 实战

CLAUDE.md 是项目的"全局记忆"，Claude 每次启动自动读取。

### 4.1 该放什么

```markdown
# 常用命令
- npm run build: 构建项目
- npm run lint: 代码检查
- npm run test -- --coverage: 运行测试并生成覆盖率报告

# 代码风格
- 使用 ES 模块（import/export），禁止 CommonJS（require）
- 组件用大驼峰，函数用小驼峰
- 常量全大写 + 下划线

# 项目规范
- 分支命名：feature/xxx、fix/xxx
- 提交信息用 conventional commits
- 每个 PR 只解决一个问题

# 重要提醒
- 代理服务端口是 9890
- 严禁修改 src/core/ 下的文件
- 每次宣称完成必须附带测试通过的证据
```

### 4.2 文件位置

- `项目根/CLAUDE.md` — 项目级，推荐检入 git 共享给团队
- `项目根/CLAUDE.local.md` — 个人项目级，加入 .gitignore
- `~/.claude/CLAUDE.md` — 全局级，所有项目通用
- 父/子目录的 CLAUDE.md 也会按需加载（monorepo 场景）

### 4.3 往 CLAUDE.md 写入规则

**方法一：会话中按 `#` 键（最快）**

在 CC 会话里直接按 `#`，输入规则，Claude 自动写入项目根目录的 CLAUDE.md：

```text
# 严禁修改 src/core/ 下的文件
# 每次宣称完成必须附带测试通过的证据
```

日常编码时顺手用 `#` 记录命令和规范，无需离开 CC。

**方法二：`/memory` 命令**

```text
/memory
```

打开编辑界面（调用 `$EDITOR` 或 `$VISUAL` 指定的编辑器），可以查看、添加、删除规则。适合批量修改。

**方法三：直接编辑文件**

项目根目录 `CLAUDE.md`，随便用什么 IDE 或编辑器打开改，格式就是普通 Markdown。

### 4.4 迭代优化

```text
会话中：边写边用 # 记录约定
里程碑时："根据本次会话进度更新 CLAUDE.md"
会话过长时：/compact 压缩上下文后继续
```

**实战演示**：新项目 `cd my-app && claude` → `/init` 自动生成基础 CLAUDE.md → 补充项目特定规范 → 开发过程中用 `#` 记录约定 → 提交到 git。

### 4.5 认识 `~/.claude/` 目录

CLAUDE.md 只是 `~/.claude/` 中的一员。这个目录是 CC 的配置和数据中心：

| 路径 | 用途 | 高频操作 |
|------|------|----------|
| `settings.json` | 全局配置（模型、环境变量、权限） | ⭐⭐⭐ 手动编辑或 `/config` |
| `skills/` | 已安装的 Skills（SKILL.md 文件） | ⭐⭐⭐ `npx skills add` 安装 |
| `plugins/` | 已安装的插件 | ⭐⭐ `/plugin install` |
| `projects/` | 按项目路径存储的配置和数据 | ⭐ 自动管理 |
| `sessions/` | 会话数据（对话历史、子代理转录） | ⭐ 自动管理，`/resume` 恢复 |
| `backups/` | 文件编辑前的备份（`/rewind` 回退依赖它） | ⭐⭐ `/rewind` 时用到 |
| `file-history/` | 文件变更历史 | ⭐ 自动管理 |
| `history.jsonl` | 当前会话的对话日志 | ⭐ 调试时查看 |
| `tasks/` | 后台任务数据（`/tasks` 列出） | ⭐⭐ `/tasks` 管理 |
| `plans/` | Plan 模式保存的方案 | ⭐ `/resume` 恢复 |
| `session-env/` | 会话级环境变量快照 | ⭐ 自动管理 |
| `shell-snapshots/` | `!` 命令的 shell 状态 | ⭐ 自动管理 |
| `ide/` | IDE 集成配置 | ⭐ `/ide` 管理 |
| `mcp-needs-auth-cache.json` | MCP OAuth 认证缓存 | ⭐ `/mcp` 管理 |

日常需要手动操作的就三个：`settings.json`（改配置）、`skills/`（装 Skills）、`backups/`（回退代码时用）。

## 5 上下文管理

### 5.1 判断时机

当看到提示 `Context left until auto-compact: 3%`，说明上下文快满了。CC 会自动触发压缩（约 150 秒），但最好主动管理。

```bash
# 随时查看上下文占用
/context
```

### 5.2 /compact vs /clear

| 操作 | 效果 | 时机 |
|------|------|------|
| `/compact` | 压缩历史，保留关键信息 | 一个任务还没完但上下文快满 |
| `/clear` | 完全重置上下文 | 任务完成，切换全新任务 |

`/clear` 后旧对话仍可通过 `/resume` 恢复，不会丢失。

### 5.3 好习惯

- 完成一个独立任务就 `/clear`，不要让多个任务堆在一个会话里
- 大任务中途用 `/compact` 续命
- 用 `/context` 定期检查，了解哪些东西在占用上下文

**实战演示**：长会话处理 bug 修复 → `/context` 发现上下文 85% 占用 → `/compact` 压缩到 40% → 继续修 → 修复完成 `/clear` → 新会话继续下一个任务。

## 6 工作流实战

### 6.1 探索 → 规划 → 编码 → 提交

```text
# 第1步：探索，不写代码
"阅读 src/auth/ 下所有文件，了解认证模块的架构。暂时不要写任何代码。"

# 第2步：制定计划（在 Plan 模式下）
"think hard，为添加双因素认证制定详细方案"

# 第3步：确认后实施
"按方案实施，每完成一部分验证后再继续"

# 第4步：提交
"提交所有变更，创建 PR，更新 CHANGELOG"
```

### 6.2 TDD 流：测试 → 失败 → 编码 → 通过

```text
# 第1步：基于预期行为编写测试
"为 validateToken 函数编写单元测试，覆盖：合法 token、过期 token、格式错误的 token。
使用 TDD 方式，不要 mock。对应的函数还不存在也没关系。"

# 第2步：确认测试失败
"运行测试，确认全部失败。不要写实现代码。"

# 第3步：提交测试
"把测试提交到 git"

# 第4步：编写实现
"编写通过所有测试的实现代码，不要修改测试。持续迭代直到全部通过。"

# 第5步：提交实现
"提交实现代码"
```

Claude 在有明确目标进行迭代时表现最好。测试用例就是最好的目标说明书。

### 6.3 截图迭代流：编码 → 截图 → 比较 → 改进

```text

# 第1步：给 Claude 设计图
"按照这个设计图实现登录页面"（粘贴截图或拖入图像）

# 第2步：截图验证
"用 Playwright 打开页面并截图，与设计图对比差异"

# 第3步：迭代改进
"修复差异，重新截图确认"

"提交所有前端变更"
```

**实战演示**：TDD 流实现一个 `formatDate` 工具函数 — 先写测试覆盖各时区和边界情况 → 测试失败 → Claude 写实现 → 测试通过 → 提交。

## 9 Skills 与子代理

### 9.1 Skills 是什么

Skills 是可复用的专业能力模板，类似给 CC 装"专业技能包"。内置 skills 和自定义 skills 都会在输入 `/` 时出现在命令菜单中。CC 也可以在任务相关时自动调用它们。

**Slash 命令与 Skill 的关系**：Slash 命令（`/xxx`）是调用入口，Skill 是实现机制。打个比方 —— 命令是菜单项的名字，Skill 是厨房里做这道菜的配方：

| | Slash 命令（调用入口） | Skill / 命令（实现） |
|------|--------------|----------------|
| 内置命令 | `/clear`, `/compact`, `/help` | CC 二进制硬编码，不可自定义 |
| 自定义命令 | `/project:xxx`, `/user:xxx` | `.claude/commands/*.md` 文件，简单 prompt 模板 |
| 内置 Skill | `/review`, `/simplify` | Anthropic 提供，以 SKILL.md 格式打包 |
| 安装的 Skill | 自动出现在 `/` 菜单 | `npx skills add` 安装到 `.claude/skills/`，可自动触发 |

Skill（新标准）比自定义命令更强大：支持 YAML frontmatter 配置（模型、工具限制、权限模式）、渐进式加载、CC 可根据场景自动触发而无需手动调用。自定义命令（`.claude/commands/`）更简单直接，适合一句就够的快捷模板。

### 9.5 创建自定义命令（两种方式）

**方式一：自定义 Slash 命令（`.claude/commands/`）** — 简单直接，适合快捷键模板。

在 `.claude/commands/` 下创建 markdown 文件，通过 `$ARGUMENTS` 接收参数。项目级放在项目根目录，全局级放在 `~/.claude/commands/`。调用方式：`/project:文件名` 或 `/user:文件名`。

**示例 1：自动修复 GitHub Issue**

创建 `.claude/commands/fix-issue.md`：

```markdown
请分析并修复 GitHub Issue：$ARGUMENTS。

步骤：
1. 使用 `gh issue view $ARGUMENTS` 获取 issue 详情
2. 理解问题描述并搜索代码库中相关文件
3. 实施必要的修复
4. 编写并运行测试验证
5. 创建描述性提交并推送 PR
6. 在 PR 描述中引用原始 issue
```

使用：`/project:fix-issue 1234` 自动处理 Issue #1234。

**示例 2：一键生成 CRUD 模块**

创建 `~/.claude/commands/gen-crud.md`：

```markdown
为 $ARGUMENTS 生成完整的 CRUD 模块，参考项目现有模式。

要求：
1. 先查看项目中已有的 CRUD 实现，遵循相同的模式和目录结构
2. 生成：路由 → 控制器 → 服务 → 数据模型 → 验证
3. 包含单元测试
4. 遵循 CLAUDE.md 中的代码规范
```

使用：`/user:gen-crud User` 一键生成 User 模块全套文件。

**示例 3：部署到 Staging**

创建 `.claude/commands/deploy-staging.md`：

```markdown
将当前分支部署到 staging 环境。

步骤：
1. 确认所有测试通过：`npm test`
2. 构建项目：`npm run build`
3. 推送到 staging 分支：`git push origin HEAD:staging`
4. 触发部署：`curl -X POST https://deploy.internal.com/api/deploy/staging`
5. 等待部署完成并检查健康状态
6. 返回部署结果和访问链接
```

使用：`/project:deploy-staging` 一键发布。

**方式二：新标准 Skill（`.claude/skills/`）** — 功能更强，支持自动触发。

用 `skills init` 创建或手动创建 `SKILL.md`（YAML frontmatter + 正文）。安装到 `.claude/skills/<name>/SKILL.md`（项目级）或 `~/.claude/skills/<name>/SKILL.md`（全局级）。调用方式：直接输入 `/skill名` 或 CC 按场景自动触发。

```bash
# 用 CLI 创建
skills init my-skill
```

生成的 `SKILL.md` 结构：

```markdown
---
name: my-skill
description: 用一句话描述这个 Skill 做什么，CC 据此决定何时自动触发
tools: Read, Bash
model: inherit
---

## 使用时机
当用户需要 X 操作时触发...

## 执行步骤
1. 先做 A
2. 再做 B
3. 验证结果
```

**两种方式对比**：

| | 自定义命令 (.commands/) | 新标准 Skill (.skills/) |
|------|--------------------|---------------------|
| 文件格式 | 纯 Markdown，无 frontmatter | SKILL.md + YAML frontmatter |
| 调用方式 | `/project:name` 或 `/user:name` | `/name` 直接调用 |
| 自动触发 | 不支持，必须手动 `/` 调用 | 支持，CC 根据 description 自动触发 |
| 配置能力 | 无（纯 prompt 模板） | 工具限制、模型选择、权限模式等 |
| 安装分发 | 手动复制文件 | `npx skills add` 一键安装 |
| 适用场景 | 简单快捷模板、个人使用 | 复杂工作流、团队共享、社区分发 |

**建议**：简单的快捷键模板用 `.claude/commands/`（方式一），需要自动触发、团队共享、或发布到社区的用 `SKILL.md`（方式二）。

### 9.7 通过 /agents 面板创建自定义子代理

这是最快捷的方式：

```text

# 2. 切换到 Library 选项卡 → 选择 Create new agent → 选择 Personal/Project
# 3. 选择 Generate with Claude，描述子代理：
"A debugging specialist that analyzes errors, identifies root causes,

recent git changes, form and test hypotheses, then verify the fix."

# 4. 选择工具（只读 → 只给 Read/Grep/Glob/Bash；可编辑 → 加上 Edit）
# 5. 选择模型（Haiku 快且便宜 / Sonnet 平衡 / inherit 继承主会话）
# 6. 保存后立即可用
```

或者手动创建 markdown 文件，项目级放 `.claude/agents/`，全局放 `~/.claude/agents/`：

```markdown

tools: Read, Edit, Bash, Grep, Glob
model: inherit
memory: project
---

You are an expert debugger.

When invoked:
1. Capture the error message and full stack trace
2. Check recent git changes for possible causes
3. Identify the root cause with evidence
4. Apply a minimal fix
5. Verify the fix works

For each issue provide: root cause → evidence → fix → verification.
```

### 9.8 子代理高级用法

**并行子代理**：多个子代理同时处理独立任务。

```text
"用三个子代理分别审查 src/auth/、src/api/、src/db/ 的代码质量，
只返回各模块的 Critical 和 Warning 级别问题"
```

**隔离模式**：子代理在独立 git worktree 中运行，文件变更不污染你的工作目录。

```yaml
---
name: refactor-worker
description: Perform isolated code refactoring tasks
tools: Read, Edit, Bash, Grep, Glob
isolation: worktree
---
```

**子代理限制特定工具**：

```yaml
# 只允许只读操作，禁止编辑文件
tools: Read, Grep, Glob, Bash

# 允许所有工具但禁止 Write 和 Edit
disallowedTools: Write, Edit

# 只允许调用特定子代理类型（作为主会话代理时）
tools: Agent(Explore, Plan), Read, Bash
```

**实战演示**：接到一个复杂的全栈 bug — "登录后偶尔跳转到 404"。主会话分析路由配置，同时派 debugger 子代理追踪 OAuth 回调链路，派 Explore 子代理搜索所有重定向逻辑。各子代理返回摘要，主会话汇总定位根因。

### 10.4 OpenSpec 实战 — Spec 驱动开发

OpenSpec 是 Fission AI 出品的 Spec-Driven Development（SDD）框架，49k+ Stars。核心理念：**先对齐 spec，再写代码**——人类和 AI 在写任何代码之前，就通过结构化 spec 文件达成共识。

**两种模式：Core vs Custom**：

OpenSpec 有两级配置——全局配置（`~/.config/openspec/config.json`）和项目配置（`openspec/config.yaml`）。全局配置中的 `profile` 字段决定你能用哪些命令：

| 模式 | 命令数 | 包含的命令 | 适用场景 |
|------|--------|-----------|----------|
| **Core**（默认） | 5 个 | `propose`、`explore`、`apply`、`sync`、`archive` | 日常快速迭代，90% 场景够用 |
| **Custom** | 11 个 | Core 的 5 个 + `new`、`continue`、`ff`、`verify`、`bulk-archive`、`onboard` | 复杂功能、需要逐步控制每个阶段 |

**模式管理命令**：

```bash
# 查看当前配置（全局 + 项目）
openspec config list

# 交互式选择 profile 和 workflows（推荐）
openspec config profile
# → 选择 core（5 个核心命令）或 custom（自选 11 个中的任意组合）

# 将配置同步到项目（每次改完 profile 都要执行）
openspec update

# 也可以直接 set
openspec config set profile core
openspec config set profile custom
```

**安装**（需要 Node.js ≥ 20.19.0）：

```bash
npm install -g @fission-ai/openspec@latest
cd your-project && openspec init
```

`openspec init` 做了三件事：创建 `openspec/` 目录结构、生成 AI 工具的 slash command 文件、初始化项目配置。支持 25+ AI 工具（Claude Code、Cursor、Windsurf、Copilot 等），不同工具的 command 语法略有差异（如 CC 用 `/opsx:propose`，Cursor 用 `/opsx-propose`）。

**完整命令清单**：

| 命令 | 所属 | 核心度 | 用途 |
|------|------|--------|------|
| `/opsx:explore [topic]` | Core | ★★★★ | 思考探索阶段：调查代码库、对比方案、评估风险，**纯只读**，不写代码 |
| `/opsx:propose <idea>` | Core | ★★★★★ | 一键创建 change + 所有规划产物（proposal + specs + design + tasks） |
| `/opsx:apply` | Core | ★★★★★ | 按 tasks.md 逐项实现，写代码、跑测试、勾选 checkbox |
| `/opsx:sync` | Core | ★★★ | 将 delta spec 合并到主 spec（长期变更的中途检查点） |
| `/opsx:archive` | Core | ★★★★ | 归档变更：合并 spec → 移动到 `archive/YYYY-MM-DD-<name>/` |
| `/opsx:new` | Custom | ★★★ | 创建空 change 目录 + `.openspec.yaml` 元数据（不生成产物） |
| `/opsx:continue` | Custom | ★★ | 按依赖图逐个创建产物（propose 一键已覆盖此流程） |
| `/opsx:ff` | Custom | ★★★ | 快速推进：一键创建所有剩余产物 |
| `/opsx:verify` | Custom | ★★★ | 三维度验证：完整性（Completeness）、正确性（Correctness）、一致性（Coherence） |
| `/opsx:bulk-archive` | Custom | ★★ | 批量归档多个 change，自动检测和解决 spec 冲突 |
| `/opsx:onboard` | Custom | ★ | 交互式新手引导，用你的实际代码库走完整流程（一次性使用） |

**评星维度：命令核心度**——缺了该命令，spec→代码→归档闭环是否还能跑通。同时综合考量命令的打磨成熟度。

**💡 按星筛选使用**：★★★★★ 是闭环支柱（`propose` 起点 + `apply` 执行），日常必用。★★★★ 是高频补充（`explore` 需求模糊时、`archive` 收尾时）。★★★ 以下按需启用。OpenSpec 通过 `npm install -g @fission-ai/openspec@latest && openspec init` 整套安装，不支持单命令安装。

**核心工作流**：

```text
# Core 默认快速路径（5 步）
/opsx:explore                   → 可选：需求不明确时先探索
/opsx:propose add-dark-mode     → 生成 proposal + specs + design + tasks
/opsx:apply                     → 按 tasks.md 逐项实现
/opsx:sync                      → 可选：长期变更中途合并 spec
/opsx:archive                   → 归档到 archive/，更新主 specs

# Custom 展开路径（逐阶段控制）
/opsx:new                       → 创建空 change 目录
/opsx:continue × N              → 逐个创建产物，步步审查
/opsx:apply                     → 逐项实现
/opsx:verify                    → 验证实现与 spec 的一致性
/opsx:archive                   → 归档

# Custom 快速路径（等同于 propose 但分开两步）
/opsx:new → /opsx:ff → /opsx:apply → /opsx:archive
```

**产生的文件结构**：

```
openspec/
├── specs/                    ← 主 spec 库（系统的"真相源"）
│   └── <domain>/
│       └── spec.md           ← 当前系统的行为规格
├── changes/                  ← 变更区（每个 change 一个文件夹）
│   └── <change-name>/
│       ├── .openspec.yaml    ← 变更元数据（schema、创建日期）
│       ├── proposal.md       ← 为什么做、做什么变更
│       ├── specs/            ← Delta spec（相对于主 spec 的增量）
│       │   └── <domain>/
│       │       └── spec.md
│       ├── design.md         ← 技术方案
│       └── tasks.md          ← 实现清单（checkbox 驱动）
├── changes/archive/          ← 归档区（按日期组织，保留审计历史）
│   └── YYYY-MM-DD-<name>/
└── config.yaml               ← 项目配置（可选）
```

**Delta Spec 机制**——OpenSpec 的核心概念。每次功能变更不直接修改主 spec，而是写一份"变更单"（Delta Spec），用三个区段标记改动内容：

```markdown
# Delta for Auth  ← 这次改的是认证模块

## ADDED Requirements     ← 新增了什么
### Requirement: Two-Factor Authentication
系统 MUST 在登录时要求第二因素验证。
#### Scenario: OTP 验证
- GIVEN 用户已开启 2FA
- WHEN 用户提交正确的凭据
- THEN 弹出 OTP 验证页面

## MODIFIED Requirements   ← 改了什么（原来的值附在后面）
### Requirement: Session Timeout
系统 SHALL 在 30 分钟无操作后过期会话。（原值：60 分钟）

## REMOVED Requirements    ← 删了什么（附带删除原因）
### Requirement: Remember Me
（因引入 2FA 而废弃）
```

一言蔽之：**Delta Spec = 规格的 git diff**。只写变化量，不重写全文。

归档时自动合并：ADDED 追加入主 spec → MODIFIED 替换对应条目 → REMOVED 从主 spec 删除。非破坏性合并，每次改动都有迹可循。

**实战演示：用 explore 探索 + propose 实施**：

```text
# 场景：做一个用户权限系统，但需求还不明确
You: /opsx:explore 用户权限系统

AI:  让我先看看你现有的代码库...
     [分析 src/ 目录结构、现有认证模块]

     1. RBAC（角色-权限-用户三层）— 最通用，生态成熟

     3. ReBAC（关系基）— 适合有层级关系的资源（如 Google Drive）

     你的场景是 B2B SaaS，多租户 + 角色分明 → 推荐 RBAC。

     需要我基于 RBAC 方案创建 change 吗？

You: 好，用 RBAC

You: /opsx:propose add-rbac-permission-system
AI:  Created openspec/changes/add-rbac-permission-system/

     ✓ specs/      — 6 个场景：管理员赋权、角色继承、权限检查、越权拒绝...
     ✓ design.md   — 中间件拦截 → PermissionService → RoleResolver 链
     ✓ tasks.md    — 12 个任务，从数据模型到集成测试

You: 审阅 proposal.md 和 design.md，发现角色继承层级太深，
     改为最多 2 层继承。更新后：

You: /opsx:apply
AI:  1.1 Role 和 Permission 数据模型 ✓
     1.2 RoleResolver 服务 ✓
     1.3 PermissionMiddleware ✓
     ...逐一完成 12 个任务...
     所有任务完成！

```

### 10.7 GSD 实战 — Spec 驱动 + 上下文工程

GSD（Get Shit Done）由 TÂCHES 团队出品，63k+ Stars。核心理念：**不要让上下文腐化**——每个开发阶段使用独立上下文窗口，解决长会话"AI 忘记前面约定"的通病。

**安装**（30 秒）：

```bash
npx get-shit-done-cc@latest
```

安装后重启 CC，GSD 的 `/gsd:*` 命令即可用。支持 Claude Code、Copilot、Gemini CLI 等 15 种运行时。不支持单命令安装。

**评星维度：防腐化必须度**——缺了该阶段/命令，上下文腐化风险多大、多快。同时综合考量执行可靠性（阶段隔离的实际效果）。

**GSD 的设计哲学："阶段隔离"**：

```
传统 CC 会话：一个上下文扛到底
  讨论 → 研究 → 计划 → 编码 → 测试 → 修bug → 讨论新功能 → 上下文爆炸...

GSD 方式：每个阶段独立上下文
  /gsd-discuss-phase → 讨论完毕，上下文重置
  /gsd-plan-phase    → 规划完毕，上下文重置
  /gsd-execute-phase → 每个 wave 用全新 200k token 上下文
  /gsd-verify-work   → 验收完毕，上下文重置
```

**项目初始化**：

| 命令 | 防腐化度 | 用途 |
|------|--------|------|
| `/gsd-new-project` | ★★★★ | 访谈→研究→需求→路线图→批准，从零初始化项目 |
| `/gsd-map-codebase` | ★★★★ | 分析现有代码库结构（已有代码的项目先跑这个） |
| `/gsd-new-milestone` | ★★★ | 开始下一个版本周期 |

**核心 6 阶段工作流**：

| 阶段 | 命令 | 防腐化度 | 职责 | 产物 |
|------|------|--------|------|------|
| **1. 讨论** | `/gsd-discuss-phase <N>` | ★★★★ | 捕获实现偏好（架构风格、API 形态、错误处理策略） | `CONTEXT.md` |
| **2. 规划** | `/gsd-plan-phase <N>` | ★★★★★ | 并行启动研究 Agent → 制定原子任务计划 → 验证计划与需求对齐 | `PLAN.md` |
| **3. 执行** | `/gsd-execute-phase <N>` | ★★★★★ | 按 wave 并行执行计划，每个 executor 200k 干净上下文，每个任务原子 commit | 代码 + 提交 |
| **4. 验证** | `/gsd-verify-work <N>` | ★★★ | 人工 UAT（用户验收测试），提取可测试交付物、逐项引导验证 | 验证报告 |
| **5. 发布** | `/gsd-ship <N>` | ★★★★ | 从已验证的阶段创建 PR，自动生成 PR 描述 | PR |
| **6. 完成** | `/gsd-complete-milestone` | ★★★ | 归档里程碑、打 release tag | tag + 归档 |

**探索与设计**：

| 命令 | 防腐化度 | 用途 |
|------|--------|------|
| `/gsd-explore` | ★★★ | 苏格拉底式探索，理清思路和方案空间 |
| `/gsd-sketch` | ★★★ | 快速草稿出方案结构 |
| `/gsd-spike` | ★★★ | 技术 spike，验证关键假设 |
| `/gsd-spec-phase <N>` | ★★★ | 独立的 spec 设计阶段 |

### 10.8 组合方案：主干 + 辅助，精简全流程

五套套装各有价值，但全装（70+ 个技能）token 开销大且功能重叠。本节给出一套经过剔重的组合方案——**以 Superpowers + OpenSpec 为主干，mattpocock + gstack 为辅助**，只保留不可替代的技能。

#### 10.8.1 设计思路

**为什么是"主干 + 辅助"结构？**

| 角色 | 套装 | 定位 |
|------|------|------|
| **主干** | Superpowers | 管"怎么做"——TDD、调试、规划，全程自动触发 |
| **主干** | OpenSpec | 管"做什么"——spec → tasks → 归档，文件系统即真相源 |
| **辅助** | mattpocock | 管"想清楚了吗"——动手前的深度追问 + Token 压缩 |
| **辅助** | gstack | 管"方向 + 审查 + 测试 + 交付"——四件事其他套装做不到 |

**剔重原则**：

1. 同类功能只留一个。比如 TDD 留 Superpowers 的（自动触发），不留 matt 的（手动调用）；调试留 Superpowers 的 `systematic-debugging`（自动触发），不留 gstack 的 `/investigate`（功能重叠）；代码审查留 gstack 的 `/review`（Staff Engineer 级别，比 Superpowers 的 `requesting-code-review` 更深入）。
2. 不可替代性是唯一标准。只有其他套装做不到的才入选——比如 `/grill-me` 的 16-50 个技术面试级追问、`/office-hours` 的 YC 式方向重构、`/qa` 的真实 Chromium 浏览器测试。
3. 低使用频率的直接砍掉。比如 Superpowers 的 `using-git-worktrees`（CC 自带 `--worktree`）、`writing-skills`（几个月用不到一次）。

**GSD 为什么没有入选？**

GSD 的"阶段隔离上下文"理念很好，但它和 OpenSpec 在 spec/plan 层严重重叠——`CONTEXT.md`/`PLAN.md` 和 `proposal.md`/`specs/`/`design.md` 会互相竞争。如果已选定 OpenSpec 作为 spec 层，GSD 就没有增量价值。如果你的场景不需要 spec 文件沉淀，可以把 OpenSpec 换成 GSD。

#### 10.8.4 归档与清理

Superpowers 和 gstack 都是整套安装，安装后需要手动归档不需要的技能。

**Superpowers 归档**（删掉 4 个目录）：

```bash
ARCHIVE_DIR=~/.claude/skills-archived/superpowers
mkdir -p "$ARCHIVE_DIR"

cd ~/.claude/skills
for skill in requesting-code-review receiving-code-review using-git-worktrees writing-skills; do
  if [ -d "$skill" ]; then
    mv "$skill" "$ARCHIVE_DIR/"
    echo "archived: $skill"
  fi
done
```

**gstack 的安装结构**（理解后才能安全归档）：

gstack 安装后有两层：

```
~/.claude/skills/
├── gstack/                              ← 仓库本体，完全不动
│   ├── SKILL.md                         ← gstack 浏览器引擎
│   ├── office-hours/SKILL.md            ← 真正的命令文件（47 个）
│   ├── qa/SKILL.md
│   ├── review/SKILL.md
│   ├── ship/SKILL.md
│   ├── browse/SKILL.md                  ← 浏览器引擎，/qa 的依赖
│   ├── <其他42个命令>/SKILL.md
│   ├── bin/ claude/ lib/ ...            ← 基础设施
│   └── ...
├── office-hours/SKILL.md → gstack/...   ← symlink（47 个，CC 通过它发现命令）
├── qa/SKILL.md → gstack/...
├── review/SKILL.md → gstack/...
├── ship/SKILL.md → gstack/...
├── browse/SKILL.md → gstack/...
├── <其他42个>/SKILL.md → ...            ← symlink，删除后命令即不可见
├── brainstorming/SKILL.md               ← Superpowers（真实文件）
├── grill-me/SKILL.md                    ← matt（真实文件）
└── ...
```

CC 不会递归扫描 `gstack/` 子目录，`./setup` 必须创建顶层 symlink 才能让 CC 发现命令。这意味着**只删 symlink 就足以让命令不可见**，`gstack/` 目录本身不需要做任何调整——避免了误删基础设施的风险。

**gstack 归档脚本**（只删 symlink）：

```bash
ARCHIVE_DIR=~/.claude/skills-archived/gstack
mkdir -p "$ARCHIVE_DIR"
KEEP="office-hours qa review ship browse"

cd ~/.claude/skills
for dir in */; do
  dir_name="${dir%/}"
  # 只归档 SKILL.md 是 symlink 的目录（即 gstack 命令）
  if [ -L "$dir_name/SKILL.md" ]; then
    case " $KEEP " in
      *" $dir_name "*) echo "keeping: $dir_name" ;;
      *) mv "$dir_name" "$ARCHIVE_DIR/" && echo "archived: $dir_name" ;;
    esac
  fi
done
```

> **关键设计**：
> - `[ -L "$dir_name/SKILL.md" ]` 只在 SKILL.md 是 symlink 时才进入归档逻辑。Superpowers、matt 的 SKILL.md 都是真实文件，自动跳过。`gstack/` 本体也因 SKILL.md 非 symlink 而跳过。
> - 不用 `continue`，用正向 `if [ -L ] then` 匹配，避免 zsh 下 `continue` + 中文注释触发解析异常（已实测验证会导致误归档全部目录）。

归档后检查：`ls ~/.claude/skills-archived/gstack/` 应有 42 个命令目录，`ls ~/.claude/skills/` 根目录只保留 5 个 gstack symlink，`~/.claude/skills/gstack/` 内容不变。

**恢复命令**（如果归档后需要恢复某个技能）：

```bash
# 从归档恢复（Superpowers）
mv ~/.claude/skills-archived/superpowers/<skill-name> ~/.claude/skills/

# 从归档恢复（gstack，只需恢复 symlink）
mv ~/.claude/skills-archived/gstack/<command-name> ~/.claude/skills/

# 单独安装（matt 技能，--skill 只写技能名即可）
npx skills add mattpocock/skills -g -a claude-code --skill <skill-name>
```

#### 10.8.5 禁用自动激活

所有手动调用的技能都应设置 `disable-model-invocation: true`，防止 Claude 在不必要时自动加载它们的 SKILL.md，浪费 token。

**需要禁用自动激活的技能**（手动 `/` 调用，不需要自动匹配触发）：

```bash
# mattpocock 的 2 个
#  ~/.claude/skills/grill-me/SKILL.md → 添加 disable-model-invocation: true
#  ~/.claude/skills/caveman/SKILL.md → 添加 disable-model-invocation: true

# gstack 保留的 5 个（含 browse 浏览器引擎），在 SKILL.md 的 frontmatter 中添加：
#  ~/.claude/skills/gstack/office-hours/SKILL.md
#  ~/.claude/skills/gstack/qa/SKILL.md
#  ~/.claude/skills/gstack/review/SKILL.md
#  ~/.claude/skills/gstack/ship/SKILL.md
#  ~/.claude/skills/gstack/browse/SKILL.md

# Superpowers 中纯手动调用的 3 个：
#  ~/.claude/skills/dispatching-parallel-agents/SKILL.md
#  ~/.claude/skills/subagent-driven-development/SKILL.md
#  ~/.claude/skills/finishing-a-development-branch/SKILL.md
```

**不能禁用自动激活的技能**（Superpowers 核心，依赖自动触发才能生效）：

```
test-driven-development    ← 检测到"写代码"自动触发 TDD 流程
systematic-debugging        ← 检测到"修 bug"自动触发四阶段调试
using-superpowers           ← 核心调度器，禁用则 Superpowers 全部失效
brainstorming               ← 检测到"设计/讨论"自动追问
writing-plans               ← 检测到"规划"自动拆解任务
executing-plans             ← 检测到批量任务自动逐项执行
verification-before-completion ← 宣称"完成"前自动检查
```

OpenSpec 的命令（`/opsx:*`）是 CLI 生成的 slash command 文件，不是 SKILL.md，天然不会被自动加载，无需处理。

**最终 Token 消耗对比**：

| | 全装（不筛选） | 本方案 |
|------|------|------|
| Superpowers | 14 个 SKILL.md 全部自动加载 | 7 个自动加载 + 3 个手动 |
| mattpocock | 19 个 | 2 个（手动，`disable-model-invocation: true`） |
| gstack | 47 个 | 5 个（手动，`disable-model-invocation: true`） |
| OpenSpec | 5 个（仅 slash command） | 5 个（仅 slash command） |
| **系统提示词中的 Skill 描述** | ~80 个 | **~7 个自动加载的短描述** |

核心收益：系统提示词里不再塞满 80 个技能的 description，只有 Superpowers 的 7 个核心技能会被自动匹配，其余都在你需要时手动 `/` 调用。

#### 10.8.6 使用场景

**场景：为已有 B2B SaaS 添加 RBAC 权限系统**

```text

You: /office-hours
AI:  [YC 式追问] 你现在的 role 字段只有 admin/user 二值？
     需要细到"某用户对某资源的某操作"吗？
```

#### 10.8.7 常见问题

**Q: 为什么不直接全装？**

全装后 Claude 的系统提示词会塞进 80+ 条 skill description，每次对话都消耗 token。更关键的是，不同套装对同类任务有不同处理方式——Superpowers 说"先写测试"，matt 也开启了 `tdd` ——Claude 不知道该听谁的。

**Q: 归档的技能以后还能用吗？**

能。归档目录按套装分了子目录（`~/.claude/skills-archived/superpowers/`、`~/.claude/skills-archived/gstack/`），文件没丢。需要时 `mv` 回原位置即可。matt 的技能不用恢复——直接 `npx skills add` 重新安装就行。

**Q: OpenSpec 能和 Superpowers 的 `writing-plans` 共存吗？**

能，而且互补。OpenSpec 的 `propose` 生成的是**静态 spec 文件**（proposal + specs + design + tasks），用于人机对齐和版本控制。Superpowers 的 `writing-plans` 是**执行时的原子步骤拆分**（2-5 分钟粒度），用于保证执行质量。一个管"目标"，一个管"怎么达成目标"。

**Q: gstack `/review` 和 Superpowers `requesting-code-review` 怎么选？**

选 gstack `/review`。理由：它是 Staff Engineer 级审查，覆盖 SQL 安全、LLM 信任边界、条件性副作用等维度；Superpowers 的版本是 5 个子代理并行审查（安全/性能/正确性/风格/测试），虽然更有仪式感但深度不如前者。PR 之前跑一次 `/review` 就够了。

## 11 并行与自动化

### 11.1 git worktree 并行会话

每个 worktree 有独立的工作目录，多个 CC 实例同时工作互不干扰。

```bash
# CC 自动创建 worktree
claude --worktree feature-auth    # 终端1：开发认证功能
claude --worktree bugfix-123      # 终端2：修复 bug

# 手动创建 worktree
git worktree add ../project-feature-a -b feature-a
cd ../project-feature-a && claude
```

**演示**：`git worktree add ../tree-a && cd ../tree-a && claude` → 终端1 修 bug → 终端2 开发新功能 → 两个 CC 并行工作。

### 11.2 无头模式批量任务

`-p` 模式适合脚本、CI 和批量处理。

```bash
# 单个任务
claude -p "分析 src/main.go 中的性能瓶颈"

# 批量处理 — 按行读取任务
cat TASK.md | while IFS= read -r line; do
    echo "Running: $line"
    timeout 180 claude -p "$line" --allowedTools "Edit"
    echo "---"
done

# CI 中自动审查
git diff main --name-only | claude -p "审查这些文件的变更，查找安全问题" --output-format json

# 批量迁移
claude -p "将 src/services/user.ts 从 Express 迁移到 Fastify" \
  --allowedTools "Edit, Bash(git commit:*)"
```

### 11.3 子代理隔离模式

自定义子代理设置 `isolation: worktree`，每次派发在独立 worktree 运行，不会出现文件冲突。

```yaml
---
name: refactor-worker
description: Perform isolated code refactoring
isolation: worktree
tools: Read, Edit, Bash, Grep
---
```

**实战演示**：`cat tasks.txt | while read t; do claude -p "$t" --allowedTools "Edit"; done` — 批量执行 10 个独立任务，每个任务自动提交。

## 12 避坑清单

### 12.1 版本控制是底线

每次让 Claude 改动前确保 git 状态干净。出了问题随时 `git reset --hard` 回到稳定点。养成原子化提交的习惯：一个功能一个 commit。

### 12.2 /clear 别省

完成一个独立任务后立即 `/clear`。堆叠多个任务在一个会话里，Claude 会混淆上下文，效率下降。

### 12.3 安全忽略文件

敏感代码不要暴露给 AI。在 `.claude/` 配置中限制访问：

```markdown
# CLAUDE.md 中加入
- 禁止读取 src/license/ 目录下的任何文件
- 禁止修改 src/core/payment/ 目录
```

### 12.4 AI 幻觉应对

识别信号：反复修改同一段代码但问题依旧、方案越来越复杂、开始建议不合理的改动。处理：`/clear` → `git reset --hard` → 重新开始，总结之前的"负面清单"告知 Claude。

### 12.5 中文设置

```markdown
# CLAUDE.md 中
- 所有对话和文档使用中文
- 注释用中文，代码标识符用英文
```

```bash
# 启动时指定
claude --language 中文
```

### 12.6 合理预期

AI 擅长 90%：CRUD、样板代码、测试用例、代码重构。剩下的 10% 需要人工介入：核心算法、架构决策、安全策略。别指望 AI 一次完美，但经过 2-3 次迭代通常会好得多。
