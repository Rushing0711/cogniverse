# 从入门到精通

## 1 模型与 Provider

Pi 支持 15+ LLM provider，不绑定特定模型，这是它与 Claude Code（仅 Anthropic）和 Codex（主推 OpenAI）的核心差异之一。

### 1.1 15+ 内置 Provider 概览

| 类别 | Provider |
|------|----------|
| 头部厂商 | Anthropic、OpenAI、Google Gemini、Amazon Bedrock、Azure OpenAI |
| 推理优化 | Groq、Cerebras、Together AI、Fireworks |
| 国内/亚洲 | DeepSeek、Kimi For Coding、MiniMax、小米 MiMo、Zai |
| 聚合器 | OpenRouter、OpenCode Zen/Go、Hugging Face |
| 其他 | xAI（Grok）、NVIDIA NIM、Cloudflare AI Gateway/Workers AI |
| 本地模型 | Ollama、vLLM、LM Studio（via `models.json` 自定义）|

### 1.2 模型 ID 格式

`provider/id`，例如：

```
anthropic/claude-sonnet-4-5
openai/gpt-4o
google/gemini-2.5-flash
deepseek/deepseek-chat
ollama/llama3.1:8b
```

```bash
# 启动时指定
$ pi --model anthropic/claude-sonnet-4-5

# 或用 --provider + --model
$ pi --provider anthropic --model claude-sonnet-4-5
```

### 1.3 :thinking 后缀简写

Pi 支持 `:thinking` 后缀简写，直接指定 thinking 级别：

```bash
$ pi --model sonnet:high "Solve this step by step"
$ pi --model opus:max "Hardest problem"
$ pi --model gpt-4o:medium "Quick task"
```

详见第 13 章 Thinking 模式。

### 1.4 模型加载优先级

从高到低：

1. `--model` / `-m` 命令行标志
2. `settings.json` 的 `defaultModel` 字段
3. 上次使用的模型
4. 按内部优先级排列的第一个可用模型

### 1.5 --models 循环切换（Ctrl+P）

配置多个模型，用 Ctrl+P 循环切换：

```bash
$ pi --models anthropic/claude-sonnet-4-5,openai/gpt-4o,google/gemini-2.5-flash
```

或在 `settings.json` 中配置：

```json
{
  "enabledModels": [
    "anthropic/claude-sonnet-4-5",
    "openai/gpt-4o",
    "google/gemini-2.5-flash"
  ]
}
```

会话内按 Ctrl+P 循环，Shift+Ctrl+P 反向循环。

### 1.6 --list-models 列出可用模型

```bash
$ pi --list-models                    # 列出所有可用模型
$ pi --list-models anthropic          # 筛选 anthropic
$ pi --list-models sonnet             # 筛选含 sonnet 的
```

### 1.7 本地模型（Ollama/vLLM/LM Studio）

通过 `~/.pi/agent/models.json` 配置本地模型：

```json
{
  "providers": {
    "ollama": {
      "baseUrl": "http://localhost:11434/v1",
      "api": "openai-completions",
      "apiKey": "ollama",
      "models": [
        { "id": "llama3.1:8b" },
        { "id": "qwen3-coder" }
      ]
    }
  }
}
```

使用：

```bash
$ pi --model ollama/qwen3-coder
```

详见第 15 章自定义 Provider 与模型。

### 1.8 OAuth 订阅登录的 provider

支持 OAuth 订阅登录的 provider：

| Provider | 订阅要求 | 说明 |
|----------|----------|------|
| ChatGPT Plus/Pro (Codex) | ChatGPT Plus/Pro | OpenAI 官方背书（Codex for OSS）|
| Claude Pro/Max | Claude Pro/Max | <span style="color:red;font-weight:bold;">第三方 harness 使用会从 extra usage 扣费，按 token 计费</span> |
| GitHub Copilot | Copilot 订阅 | 支持 GitHub Enterprise Server |

在 TUI 中运行 `/login` 选择对应 provider 完成 OAuth。

### 1.9 凭证解析顺序

Pi 按以下顺序解析凭证：

1. CLI `--api-key` 标志
2. `~/.pi/agent/auth.json` 中存储的 key
3. 环境变量（如 `ANTHROPIC_API_KEY`）
4. `models.json` 自定义 provider 的 keys

```bash
# 1. CLI 标志（最高优先级）
$ pi --api-key sk-xxx --provider anthropic

# 2. auth.json（/login 后存储）
$ pi   # 自动读取 auth.json

# 3. 环境变量
$ export ANTHROPIC_API_KEY=sk-ant-xxx
$ pi

# 4. models.json 自定义 provider 的 apiKey 字段
```

## 2 项目信任与安全

Pi 的安全模型与 Claude Code/Codex/OpenCode 不同--无内置权限弹窗或沙箱，通过项目信任机制 + 容器化方案实现安全。

### 2.1 Pi 的安全模型

<span style="color:red;font-weight:bold;">Pi 不内置权限弹窗或沙箱</span>。内置工具以 pi 进程权限运行，可读写用户可写的文件、执行 shell 命令。

核心理念：提供原语，而非功能。权限控制通过以下方式实现：

- **项目信任机制**：防止仓库静默更改 pi 设置/扩展
- **容器化**：对不可信仓库/代码用容器运行
- **extension 自建权限门**：如 `permission-gate.ts` 示例

### 2.2 项目信任机制

当项目含以下资源时，交互式启动会询问是否信任：

| 资源 | 说明 |
|------|------|
| `.pi/settings.json` | 项目设置 |
| `.pi/extensions` | 项目扩展 |
| `.pi/skills` | 项目 skills |
| `.pi/SYSTEM.md` | 项目系统提示 |
| `.agents/skills` | Agent 标准 skills |

信任后才能加载项目设置和扩展。

### 2.3 /trust 命令

```
/trust
```

保存项目信任决策到 `~/.pi/agent/trust.json`，重启后生效。

### 2.4 defaultProjectTrust

在 `settings.json` 中配置默认信任行为：

```json
{
  "defaultProjectTrust": "ask"    // ask（默认）| always | never
}
```

| 值 | 行为 |
|----|------|
| `ask` | 每次询问 |
| `always` | 自动信任所有项目 |
| `never` | 自动拒绝所有项目 |

### 2.5 --approve / --no-approve 单次覆盖

```bash
$ pi -a          # --approve，单次批准项目信任
$ pi -na         # --no-approve，单次拒绝
```

适合非交互模式或 CI 场景。

### 2.6 容器化方案概览

对不可信仓库/代码，官方提供三种容器化方案：

| 方案 | 说明 | 要求 |
|------|------|------|
| **Gondolin** | 本地 Linux micro-VM，host pi + 工具路由进 VM | Node.js >= 23.6.0 + QEMU |
| **Plain Docker** | 整个 pi 进程跑在 Docker | Docker |
| **OpenShell**（NVIDIA）| 策略控制沙箱，支持本地/远程网关 | OpenShell |

### 2.7 对不可信仓库/代码的建议

- 用容器（Gondolin/Docker/OpenShell）运行
- 不要在不可信仓库中启用 `defaultProjectTrust: "always"`
- 用 `--no-approve` 单次拒绝
- 检查 `.pi/extensions` 和 `.pi/skills` 的内容

### 2.8 extension 自建权限门

通过 extension 自建权限确认流程，官方提供示例：

- `examples/extensions/permission-gate.ts`：工具调用前询问确认
- `examples/extensions/protected-paths.ts`：保护特定路径不被修改

这些 extension 监听 `tool_call` 事件，在工具执行前拦截并询问用户。详见第 10 章 Extensions 开发。

## 3 基本交互（TUI）

### 3.1 启动与四区界面

```bash
$ pi                    # 在当前目录启动 TUI
$ pi /path/to/project   # 指定工作目录
$ pi "explain this codebase"   # 带初始 prompt 启动
```

Pi 的 TUI 分四区：

| 区域 | 说明 |
|------|------|
| 启动头部 | 快捷键、已加载上下文文件、prompt 模板、skills、extensions |
| 消息区 | 用户消息、assistant 响应、工具调用/结果、通知、错误、扩展 UI |
| 编辑器 | 输入区，边框颜色 = thinking 级别 |
| 页脚 | 工作目录、会话名、token/cache 用量、成本、上下文用量、当前模型 |

### 3.2 编辑器功能

| 功能 | 快捷键 | 说明 |
|------|--------|------|
| 文件引用 | `@` | fuzzy 搜索文件，引用内容 |
| 路径补全 | `Tab` | 自动补全路径 |
| 多行输入 | `Shift+Enter` | 换行不提交（Windows Terminal 用 `Ctrl+Enter`）|
| 粘贴图片 | `Ctrl+V` | 粘贴图片/文本（Windows 用 `Alt+V`），或拖拽图片 |
| 外部编辑器 | `Ctrl+G` | 打开 `$VISUAL`/`$EDITOR` 编写长 prompt |

### 3.3 !command 与 !!command

在编辑器中输入 `!` 开头执行 shell 命令：

| 语法 | 说明 |
|------|------|
| `!command` | 运行 shell 并发送输出给模型 |
| `!!command` | 运行 shell 但不发送输出给模型 |

示例：

```
基于 !git log --oneline -10 的最近提交，生成 release notes

!!mkdir -p /tmp/work    # 创建目录但不告诉模型
```

### 3.4 Ctrl+G 外部编辑器

按 `Ctrl+G` 打开外部编辑器编写长 prompt：

```json
// ~/.pi/agent/settings.json
{
  "externalEditor": "code --wait"    // VS Code（需 --wait）
}
```

支持的编辑器：

```json
{ "externalEditor": "code --wait" }       // VS Code
{ "externalEditor": "cursor --wait" }     // Cursor
{ "externalEditor": "vim" }
{ "externalEditor": "nano" }
```

未配置时回退到 `$VISUAL` / `$EDITOR` 环境变量（Windows Notepad，其他 nano）。

### 3.5 消息队列 steer/follow-up

Pi 独特的消息队列机制，agent 工作时可排队消息：

| 快捷键 | 说明 |
|--------|------|
| `Enter`（agent 工作时）| 排队 steering 消息，当前 turn 工具执行完后投递 |
| `Alt+Enter` | 排队 follow-up 消息，agent 完成所有工作后投递 |
| `Escape` | 中止并恢复排队消息到编辑器 |
| `Alt+Up` | 取回排队消息 |

**steering vs follow-up**：

- **steering**：加入当前 turn，立即影响正在进行的推理
- **follow-up**：排到下一 turn，当前 turn 完成后再处理

使用场景：

```
# Codex 正在改错文件
别动 src/core/，改 src/utils/    # Enter 发送 steering

# Codex 正在写测试
做完后顺便跑一遍 npm test        # Alt+Enter 发送 follow-up
```

### 3.6 页脚信息

页脚显示实时状态：

| 信息 | 说明 |
|------|------|
| 工作目录 | 当前 cwd |
| 会话名 | `/name` 设置的名称 |
| token 用量 | ↑输入、↓输出 |
| cache 用量 | R缓存读、W缓存写、CH缓存命中率 |
| 成本 | 本次会话费用 |
| 上下文用量 | 已用/总量 |
| 当前模型 | provider/model |

### 3.7 好 Prompt 公式与修正方向

清晰的 prompt 结构：

```
[上下文] + [任务] + [约束] + [期望输出格式]
```

示例：

```
我在做一个 VitePress 知识库（见 docs/.vitepress/config.mjs），
需要新增 Pi 文档目录 docs/aillm/Pi/。
约束：仿照 docs/aillm/OpenCode/ 的风格，中文为主，代码标识符英文。
请先列出 00-引言.md 的章节大纲，不要直接写代码。
```

**修正方向**：如果 Pi 走偏了，用 `Escape` 中止当前 turn，或用 `Enter` 发送 steering 消息纠正：

```
# Pi 正在过度抽象
停，不要引入 BaseAgent 抽象层，直接在现有函数上改

# Pi 正在改错文件
这个改动应该在 src/utils.ts，不是 src/core.ts
```

## 4 AGENTS.md 与上下文工程

Pi 兼容 `AGENTS.md` 和 `CLAUDE.md`，并提供精细的上下文控制。

### 4.1 AGENTS.md 该放什么

```markdown
# 项目名称

## 常用命令
- `pnpm docs:dev` - 启动开发服务器
- `pnpm docs:build` - 构建生产版本

## 代码风格
- 文档使用中文
- 代码标识符保持英文
- 配置示例用 JSON

## 项目规范
- 一定不要删除文档 *.md 中的图片
- 构建产物 docs/.vitepress/dist 不提交到 git

## 重要提醒
- 包管理器必须用 pnpm
- 端口 8750 用于本地开发
```

### 4.2 兼容 CLAUDE.md 回退

Pi 兼容 Claude Code 的指令文件：

- 优先读取 `AGENTS.md`
- 若不存在，回退读取 `CLAUDE.md`

从 Claude Code 迁移到 Pi 时，<span style="color:red;font-weight:bold;">无需立即重命名 CLAUDE.md</span>，Pi 自动兼容。

### 4.3 SYSTEM.md 替换默认系统提示

`~/.pi/agent/SYSTEM.md` 或项目级 `.pi/SYSTEM.md` **完全替换** Pi 的默认系统提示：

```markdown
<!-- ~/.pi/agent/SYSTEM.md -->
You are a senior code reviewer. Focus on:
- Security vulnerabilities
- Performance issues
- Code quality

Always cite file paths and line numbers.
```

<span style="color:red;font-weight:bold;">注意：替换默认系统提示会失去 Pi 内置的编码能力引导，慎用</span>。

### 4.4 APPEND_SYSTEM.md 追加系统提示

`~/.pi/agent/APPEND_SYSTEM.md` 或项目级 `.pi/APPEND_SYSTEM.md` **追加**到默认系统提示：

```markdown
<!-- ~/.pi/agent/APPEND_SYSTEM.md -->
## Additional instructions
- Always run `npm test` after modifying JavaScript files.
- Prefer `pnpm` when installing dependencies.
- Ask for confirmation before adding new production dependencies.
```

推荐用 `APPEND_SYSTEM.md` 而非 `SYSTEM.md`，保留 Pi 内置能力。

### 4.5 --no-context-files / -nc

禁用所有上下文文件（AGENTS.md/CLAUDE.md/SYSTEM.md/APPEND_SYSTEM.md）：

```bash
$ pi -nc
$ pi --no-context-files
```

适合临时测试，不加载任何项目指令。

### 4.6 --system-prompt 与 --append-system-prompt

命令行覆盖：

```bash
# 完全替换系统提示
$ pi --system-prompt "You are a code reviewer"

# 追加到系统提示
$ pi --append-system-prompt "Focus on security"
```

适合一次性任务，不修改配置文件。

### 4.7 全局 vs 项目级指令

| 层级 | 位置 | 用途 |
|------|------|------|
| 全局 | `~/.pi/agent/AGENTS.md` | 用户全局偏好 |
| 项目 | `<repo>/AGENTS.md` 或 `<repo>/CLAUDE.md` | 项目特定规则 |
| 全局追加 | `~/.pi/agent/APPEND_SYSTEM.md` | 全局追加系统提示 |
| 项目追加 | `<repo>/.pi/APPEND_SYSTEM.md` | 项目追加系统提示 |
| 全局替换 | `~/.pi/agent/SYSTEM.md` | 全局替换系统提示 |
| 项目替换 | `<repo>/.pi/SYSTEM.md` | 项目替换系统提示 |

### 4.8 Context engineering 理念

Pi 强调 **context engineering**（上下文工程）：

- **最小系统提示**：Pi 的默认系统提示很短，避免占用过多上下文
- **精细控制**：通过 `SYSTEM.md` / `APPEND_SYSTEM.md` / `--system-prompt` 精细控制
- **按需加载**：Skills 的渐进式披露，完整指令按需加载
- **Compaction**：自动压缩旧上下文，保留关键信息

核心理念：上下文是稀缺资源，应该精确控制每一 token 的用途。

## 5 settings.json 配置体系

### 5.1 配置文件位置

| 位置 | 作用 |
|------|------|
| `~/.pi/agent/settings.json` | 全局设置 |
| `.pi/settings.json` | 项目设置（覆盖全局，嵌套对象合并）|

### 5.2 项目设置覆盖全局

项目设置覆盖全局设置，嵌套对象深度合并：

```json
// ~/.pi/agent/settings.json（全局）
{
  "defaultModel": "anthropic/claude-sonnet-4-5",
  "compaction": {
    "enabled": true,
    "reserveTokens": 16384
  }
}
```

```json
// .pi/settings.json（项目）
{
  "compaction": {
    "reserveTokens": 32768    // 仅覆盖这个字段，enabled 仍为 true
  }
}
```

## 6 pi -p 非交互模式

`pi -p` 是 Pi 的非交互式入口，适合脚本化、CI/CD、批量处理。与 `codex exec` 或 `claude -p` 类似。

### 6.1 基本用法

```bash
# 流式输出到 stdout
$ pi -p "summarize the repository structure"

# 输出重定向到文件
$ pi -p "generate release notes for the last 10 commits" | tee release-notes.md
```

### 6.2 管道输入

```bash
$ cat README.md | pi -p "Summarize this"
$ echo "explain async/await" | pi -p
```

### 6.3 @ 文件引用

```bash
$ pi -p @README.md "总结这个文件"
$ pi -p @screenshot.png "图里是什么"
$ pi -p @src/index.ts @src/utils.ts "compare these files"
```

可重复 `@` 引用多个文件作为上下文。

### 6.4 --mode json

输出所有事件为 JSON 行（JSONL）：

```bash
$ pi --mode json "summarize the repo structure" | jq
```

适合用 `jq` 过滤特定事件，或传递给下游程序处理。

### 6.5 --mode rpc

RPC 模式，通过 stdin/stdout JSONL 通信：

```bash
$ pi --mode rpc
# 等待 stdin 输入 JSON 指令，输出 JSON 响应
```

适合构建自定义 UI 或集成到其他应用。详见第 16 章编程式使用。

### 6.6 --export 导出 HTML

## 7 会话管理与树形分支

Pi 的会话管理独特之处是**树形结构**--单文件存所有分支，`/tree` 可原地探索替代方案。

### 7.1 JSONL 树结构

会话存为 JSONL 文件，每条目有 `id` 和 `parentId`，构成树形结构：

```jsonl
{"id":"msg-1","parentId":null,"type":"user","content":"解释这个函数"}
{"id":"msg-2","parentId":"msg-1","type":"assistant","content":"..."}
{"id":"msg-3","parentId":"msg-2","type":"user","content":"换种实现"}
{"id":"msg-4","parentId":"msg-3","type":"assistant","content":"..."}
{"id":"msg-5","parentId":"msg-2","type":"user","content":"还是用第一种，但优化下"}
{"id":"msg-6","parentId":"msg-5","type":"assistant","content":"..."}
```

所有分支在一个文件里，`/tree` 可跳转到任意节点继续。

### 7.2 会话文件位置

```
~/.pi/agent/sessions/--<path>--/<timestamp>_<uuid>.jsonl
```

按工作目录组织，例如：

```
~/.pi/agent/sessions/--Users-wenqiu-WebstormProjects-cogniverse--/2026-07-11_10-30-00_abc123.jsonl
```

## 8 Compaction 上下文压缩

Pi 自动压缩旧上下文，保留关键信息，避免上下文膨胀。

### 8.1 自动压缩触发条件

当上下文接近上限时自动触发压缩：

```
contextTokens > contextWindow - reserveTokens
```

- `contextTokens`：当前上下文 token 数
- `contextWindow`：模型上下文窗口
- `reserveTokens`：保留缓冲区（默认 16384）

### 8.2 /compact [instructions] 手动压缩

```
/compact
/compact 重点保留 Git 相关的讨论
```

手动触发压缩，可带 instructions 指定压缩重点。

### 8.3 compaction 配置字段

```json
{
  "compaction": {
    "enabled": true,            // 默认 true
    "reserveTokens": 16384,     // 触发阈值缓冲
    "keepRecentTokens": 20000   // 保留最近 token 数
  }
}
```

| 字段 | 说明 |
|------|------|
| `enabled` | 是否启用自动压缩 |
| `reserveTokens` | 保留缓冲区，超过 `contextWindow - reserveTokens` 触发 |
| `keepRecentTokens` | 压缩时保留最近这么多 token 不压缩 |

### 8.4 split turn

单 turn 超过预算时，Pi 支持 **split turn**（mid-turn 切割）：

- 在 turn 执行过程中检测到超预算
- 中途暂停，压缩旧上下文
- 恢复 turn 继续

避免单 turn 过长导致失败。

### 8.5 分支摘要

切换分支时，Pi 可生成分支摘要：

```json
{
  "branchSummary": {
    "reserveTokens": 8000,
    "skipPrompt": "Summarize the work done in this branch."
  }
}
```

- `reserveTokens`：分支摘要占用的 token 预算
- `skipPrompt`：生成摘要的 prompt

分支摘要在切换分支时作为上下文，避免丢失前文。

### 8.6 扩展自定义 compaction

Extensions 可订阅 `session_before_compact` 和 `session_compact` 事件，自定义 compaction 逻辑：

```typescript
import type { Extension } from "@earendil-works/pi-coding-agent"

export const MyCompaction: Extension = async ({ pi }) => {
  pi.on("session_before_compact", (event) => {
    // 自定义压缩前处理
  })

  pi.on("session_compact", (event) => {
    // 自定义压缩后处理
  })
}
```

详见第 10 章 Extensions 开发。

### 8.7 /compact vs 自动压缩

| 操作 | 触发时机 | 适用场景 |
|------|----------|----------|
| 自动压缩 | 上下文接近上限 | 日常使用，无需干预 |
| `/compact` | 手动触发 | 上下文变大但任务延续，想主动压缩 |
| `/new` | 手动触发 | 完全切换到不相关的任务 |

何时手动 `/compact`：

- 上下文用量 > 70%
- Pi 开始"忘事"
- 切换任务方向，但想保留部分上下文

## 9 Skills

Pi 实现 [Agent Skills 标准](https://agentskills.io)，与 Claude Code/Codex/OpenCode 的 skills 兼容。

### 9.1 Agent Skills 标准

Agent Skills 是开放标准，定义了 skill 的格式和加载机制。Pi 实现该标准，意味着：

- Pi 的 skills 可被其他兼容 harness 使用
- Claude Code/Codex/OpenCode 的 skills 可被 Pi 使用
- 跨 harness 复用，无需迁移

### 9.2 SKILL.md frontmatter

```markdown
---
name: git-release
description: Create consistent releases and changelogs
license: MIT
compatibility: pi
metadata:
  audience: maintainers
  workflow: github
allowed-tools:
  - read
  - bash
disable-model-invocation: false
---

## What I do
- Draft release notes from merged PRs
- Propose a version bump
```

| 字段 | 说明 |
|------|------|
| `name`（必填）| 1-64 字符，小写字母数字+连字符，正则 `^[a-z0-9]+(-[a-z0-9]+)*$` |
| `description`（必填）| 最长 1024 字符 |
| `license`（可选）| 许可证 |
| `compatibility`（可选）| 兼容性标记 |
| `metadata`（可选）| 字符串映射 |
| `allowed-tools`（可选）| 允许使用的工具列表 |
| `disable-model-invocation`（可选）| 禁止模型自动调用 |

### 9.3 加载位置

Pi 从以下位置加载 skills：

| 范围 | 位置 |
|------|------|
| 全局 | `~/.pi/agent/skills/` |
| 全局（兼容）| `~/.agents/skills/` |
| 项目（信任后）| `.pi/skills/` |
| 项目（兼容）| `.agents/skills/`（从 cwd 向上到 git 根）|
| Packages | 通过 `pi install` 安装的包 |
| settings | `settings.json` 的 `skills` 字段 |
| CLI | `--skill <path>` |

### 9.4 渐进式披露机制

Pi 采用**渐进式披露**（progressive disclosure）：

1. **启动时**：扫描所有 skill，提取 `name` + `description` 注入系统提示
2. **任务匹配时**：模型用 `read` 工具加载完整 `SKILL.md`
3. **执行指令**：用相对路径引用 skill 目录内的脚本/资源

只有 `description` 始终在上下文，完整指令按需加载，<span style="color:red;font-weight:bold;">不破坏 prompt cache</span>。

### 9.5 跨 harness 兼容

复用 Claude Code/Codex 的 skills，在 `settings.json` 中添加路径：

```json
{
  "skills": [
    "~/.claude/skills",
    "~/.codex/skills"
  ]
}
```

Pi 会自动扫描这些目录，加载所有 `SKILL.md`。

### 9.6 /skill:name 调用

```
/skill:git-release
/skill:git-release v2.1.0
```

显式调用 skill，可带参数。

### 9.7 enableSkillCommands

```json
{
  "enableSkillCommands": true    // 默认 true
}
```

设为 `false` 禁用 `/skill:name` 命令，但 skills 仍可被模型自动调用。

### 9.8 自动调用 vs 显式调用

| 调用方式 | 说明 |
|----------|------|
| 自动调用 | 模型根据 `description` 自动加载并执行 |
| 显式调用 | `/skill:name` 手动触发 |

自动调用依赖 `description` 的质量，写好 description 是关键。

## 10 Extensions 开发

Extensions 是 Pi 的核心扩展机制，用 TypeScript 编写，可监听丰富的生命周期事件。

### 10.1 Extension 是什么

Extension 是 TypeScript 模块，通过 jiti 加载，**无需编译**：

```typescript
// ~/.pi/agent/extensions/my-extension.ts
import type { Extension } from "@earendil-works/pi-coding-agent"

export const MyExtension: Extension = async (ctx) => {
  const { pi } = ctx
  // 订阅事件、注册工具/命令/快捷键等
}
```

### 10.2 扩展位置

| 位置 | 作用 |
|------|------|
| `~/.pi/agent/extensions/*.ts` | 全局扩展 |
| `.pi/extensions/*.ts` | 项目扩展（需信任）|
| `~/.pi/agent/extensions/<name>/index.ts` | 子目录形式 |

支持 `pi -e ./path.ts` 临时测试，放自动发现目录可 `/reload` 热重载。

### 10.3 生命周期事件流

Pi 的生命周期事件按顺序触发：

```
project_trust -> session_start -> resources_discover
-> input -> before_agent_start -> agent_start
-> turn_start -> context -> before_provider_headers
-> before_provider_request -> after_provider_response
-> tool_execution_start -> tool_call（可 block）-> tool_execution_update
-> tool_result（可修改）-> tool_execution_end -> turn_end
-> agent_end -> agent_settled
```

### 10.4 事件分类

| 类别 | 事件 |
|------|------|
| 资源 | `project_trust`、`resources_discover` |
| 会话 | `session_start`、`session_info_changed`、`session_before_switch`、`session_before_fork`、`session_before_compact`/`session_compact`、`session_before_tree`/`session_tree`、`session_shutdown` |
| Agent | `before_agent_start`（可注入消息/改系统提示）、`agent_start`/`agent_end`/`agent_settled`、`turn_start`/`turn_end`、`message_start`/`message_update`/`message_end`、`tool_execution_*` |
| 上下文 | `context`（每次 LLM 调用前，可修改消息）|
| Provider | `before_provider_headers`、`before_provider_request`、`after_provider_response` |
| 模型 | `model_select`、`thinking_level_select` |
| 工具 | `tool_call`（可 block、可修改 `event.input`）、`tool_result`（链式中间件，可修改结果）|
| 用户 bash | `user_bash`（可拦截 `!`/`!!` 命令）|
| 输入 | `input`（可 transform/handle/continue）|

### 10.5 pi.on(event, handler) 订阅

```typescript
import type { Extension } from "@earendil-works/pi-coding-agent"

export const MyExtension: Extension = async (ctx) => {
  const { pi } = ctx

  pi.on("session_start", (event) => {
    console.log("Session started:", event.sessionId)
  })

  pi.on("tool_call", (event) => {
    if (event.tool === "bash") {
      const cmd = event.input.command
      if (cmd.includes("rm -rf /")) {
        return { block: true, reason: "Destructive command blocked" }
      }
    }
  })

  pi.on("session.idle", (event) => {
    // 会话完成时通知
  })
}
```

### 10.15 实战示例

#### permission-gate.ts

```typescript
import type { Extension } from "@earendil-works/pi-coding-agent"

export const PermissionGate: Extension = async (ctx) => {
  const { pi, ui } = ctx

  pi.on("tool_call", async (event) => {
    if (event.tool === "bash") {
      const cmd = event.input.command
      const approved = await ui.confirm(`Run: ${cmd}?`, false)
      if (!approved) {
        return { block: true, reason: "User denied" }
      }
    }
  })
}
```

#### protected-paths.ts

```typescript
import type { Extension } from "@earendil-works/pi-coding-agent"

const PROTECTED = [".env", "secrets.json", "*.key"]

export const ProtectedPaths: Extension = async (ctx) => {
  const { pi, ui } = ctx

  pi.on("tool_call", async (event) => {
    if (event.tool === "edit" || event.tool === "write") {
      const path = event.input.path
      if (PROTECTED.some(p => path.includes(p))) {
        return { block: true, reason: `Protected path: ${path}` }
      }
    }
  })
}
```

## 13 Thinking 模式

Pi 支持 7 级 thinking 级别，控制模型推理深度。

### 13.1 7 级别

| 级别 | 说明 | 适用场景 |
|------|------|----------|
| `off` | 不思考 | 简单任务、最低延迟 |
| `minimal` | 最低思考 | 快速响应 |
| `low` | 低思考 | 简单 well-scoped 任务 |
| `medium` | 中等思考（默认）| 平衡速度与深度 |
| `high` | 高思考 | 复杂逻辑、边界条件 |
| `xhigh` | 极高思考 | 极复杂推理 |
| `max` | 最大思考 | 最难问题 |

### 13.2 Shift+Tab 循环

```
Shift+Tab
```

循环切换 thinking 级别：off -> minimal -> low -> medium -> high -> xhigh -> max -> off...

### 13.3 --thinking 命令行

```bash
$ pi --thinking high "solve this step by step"
$ pi --thinking max "hardest problem"
$ pi --thinking off "quick question"
```

### 13.4 defaultThinkingLevel 配置

```json
{
  "defaultThinkingLevel": "medium"
}
```

设置默认 thinking 级别，启动时自动应用。

### 13.5 模型简写

`--model` 支持 `:thinking` 后缀：

```bash
$ pi --model sonnet:high "Solve this"
$ pi --model opus:max "Hardest problem"
$ pi --model gpt-4o:medium "Quick task"
```

等同于 `--model sonnet --thinking high`。

### 13.6 thinkingBudgets

为每级设置 token 预算：

```json
{
  "thinkingBudgets": {
    "minimal": 2000,
    "low": 4000,
    "medium": 8000,
    "high": 16000,
    "xhigh": 32000,
    "max": 64000
  }
}
```

预算越高，模型思考越深入，但消耗更多 token。

### 13.7 hideThinkingBlock

```json
{
  "hideThinkingBlock": false    // 默认 false
}
```

设为 `true` 隐藏思考块，只显示最终答案。思考过程仍在后台进行。

### 13.8 Ctrl+T 折叠/展开

```
Ctrl+T
```

折叠或展开当前消息的 thinking 块。

### 13.9 边框颜色指示

编辑器边框颜色指示当前 thinking 级别：

| 级别 | 颜色（默认主题）|
|------|-----------------|
| off | 灰色 |
| minimal | 深灰 |
| low | 蓝色 |
| medium | 青色 |
| high | 黄色 |
| xhigh | 橙色 |
| max | 红色 |

颜色可在主题 JSON 中自定义（见第 14 章）。

### 13.10 模型限制

<span style="color:red;font-weight:bold;">非 reasoning 模型强制 `off`</span>，不同模型支持的级别不同。

通过 `thinkingLevelMap` 配置模型支持的级别：

```json
{
  "providers": {
    "openai": {
      "models": {
        "gpt-4o": {
          "thinkingLevelMap": {
            "off": "off",
            "low": "low",
            "medium": "medium",
            "high": "high"
          }
        }
      }
    }
  }
}
```

```

# 选择级别后立即生效
```

## 15 自定义 Provider 与模型

Pi 通过 `models.json` 配置自定义 provider，或通过 extension 实现完全自定义。

### 15.1 models.json 文件位置

`~/.pi/agent/models.json`：

```json
{
  "providers": {
    "my-provider": {
      "baseUrl": "https://api.my-provider.com/v1",
      "api": "openai-completions",
      "apiKey": "{env:MY_API_KEY}",
      "models": [
        {
          "id": "my-model",
          "name": "My Model Display Name",
          "contextWindow": 200000,
          "maxTokens": 65536
        }
      ]
    }
  }
}
```

### 15.2 providers 配置

| 字段 | 说明 |
|------|------|
| `baseUrl` | API 端点 |
| `api` | API 类型（见 15.3）|
| `apiKey` | API Key（支持 `!command`、`$ENV_VAR`）|
| `models` | 模型列表 |

### 15.3 API 类型

| API 类型 | 说明 |
|----------|------|
| `openai-completions` | OpenAI Chat Completions API（`/v1/chat/completions`）|
| `openai-responses` | OpenAI Responses API（`/v1/responses`）|
| `anthropic-messages` | Anthropic Messages API |
| `google-generative-ai` | Google Generative AI API |

### 15.4 模型字段

```json
{
  "id": "llama3.1:8b",
  "name": "Llama 3.1 8B",
  "reasoning": false,
  "input": ["text", "image"],
  "contextWindow": 128000,
  "maxTokens": 4096,
  "cost": {
    "input": 0.0,
    "output": 0.0
  },
  "compat": {
    "tools": true,
    "streaming": true
  },
  "thinkingLevelMap": {
    "off": "off"
  }
}
```

| 字段 | 说明 |
|------|------|
| `id` | 模型 ID |
| `name` | 显示名称 |
| `reasoning` | 是否支持 reasoning |
| `input` | 支持的输入类型（text/image）|
| `contextWindow` | 上下文窗口 |
| `maxTokens` | 最大输出 token |
| `cost` | 费用（input/output per token）|
| `compat` | 兼容性（tools/streaming）|
| `thinkingLevelMap` | 支持的 thinking 级别映射 |

### 15.5 覆盖内置 provider

在 `models.json` 中定义与内置同名的 provider，会覆盖内置：

```json
{
  "providers": {
    "anthropic": {
      "baseUrl": "https://my-proxy.example.com/v1",
      "api": "anthropic-messages",
      "apiKey": "{env:ANTHROPIC_API_KEY}",
      "models": [...]
    }
  }
}
```

### 15.6 modelOverrides per-model 覆盖

只覆盖特定模型的字段，不整个替换 provider：

```json
{
  "modelOverrides": {
    "anthropic/claude-sonnet-4-5": {
      "contextWindow": 1000000,
      "cost": {
        "input": 0.003,
        "output": 0.015
      }
    }
  }
}
```

### 15.7 Ollama 配置示例

```json
{
  "providers": {
    "ollama": {
      "baseUrl": "http://localhost:11434/v1",
      "api": "openai-completions",
      "apiKey": "ollama",
      "models": [
        { "id": "llama3.1:8b", "name": "Llama 3.1 8B" },
```


### 15.8 extension registerProvider

通过 extension 实现完全自定义 provider（含 OAuth、自定义流式）：

```typescript
import type { Extension } from "@earendil-works/pi-coding-agent"

export const MyProvider: Extension = async (ctx) => {
  const { pi } = ctx

  pi.registerProvider("my-provider", {
    name: "My Provider",
    baseUrl: "https://api.my-provider.com/v1",
    api: "openai-completions",
    // 支持 OAuth 流程
    oauth: {
      authorizeUrl: "https://api.my-provider.com/oauth/authorize",
      tokenUrl: "https://api.my-provider.com/oauth/token",
      clientId: "{env:MY_CLIENT_ID}",
      scope: "read write"
    },
    // 自定义请求头
    headers: {
      "X-Custom-Header": "value"
    },
    // 自定义流式处理
    streamParser: async (chunk) => {
      // 解析自定义流式格式
      return { content: "..." }
    }
  })
}
```

### 15.9 应用场景

- 接入公司内部 LLM 代理
- 接入自定义训练的模型
- 实现 OAuth 认证的 provider（如企业 SSO）
- 接入非标准 API（自定义流式格式）

```typescript
import {
```

### 17.5 迁移指南

#### CLAUDE.md -> AGENTS.md（兼容回退，无需立即迁移）

Pi 兼容 `CLAUDE.md`，无需立即重命名：

```bash
# 保持现状，Pi 自动回退读取 CLAUDE.md
$ pi

# 或重命名（推荐，长期迁移）
$ mv CLAUDE.md AGENTS.md
```

#### skills 跨 harness 复用（Pi 实现 Agent Skills 标准）

Pi 兼容 Claude Code/Codex/OpenCode 的 skills，在 `settings.json` 中添加路径：

```json
{
  "skills": [
    "~/.claude/skills",
    "~/.codex/skills",
    "~/.config/opencode/skills"
  ]
}
```

#### hooks -> extensions（TypeScript 重构）

Pi 没有独立 hooks 系统，需重构为 extension：

```typescript
// .pi/extensions/my-hooks.ts
import type { Extension } from "@earendil-works/pi-coding-agent"

export const MyHooks: Extension = async (ctx) => {
  const { pi } = ctx

  // 原 PreToolUse hook 逻辑
  pi.on("tool_call", (event) => {
    if (event.tool === "bash") {
      // 拦截逻辑
    }
  })

  // 原 PostToolUse hook 逻辑
  pi.on("tool_result", (event) => {
    // 后处理逻辑
  })

  // 原 Stop hook 逻辑
  pi.on("session.idle", (event) => {
    // 通知逻辑
  })
}
```

事件映射：

| Claude Code | Pi |
|-------------|-----|
| `PreToolUse` | `tool_call` |
| `PostToolUse` | `tool_result` |
| `Stop` | `session.idle` |
| `SessionStart` | `session_start` |
| `UserPromptSubmit` | `input` |

#### permissions -> 容器或 extension

Pi 没有内置权限系统，替代方案：

1. **容器化**：用 Gondolin/Docker/OpenShell 运行不可信代码
2. **extension 权限门**：用 `permission-gate.ts` 或 `protected-paths.ts` 示例
3. **项目信任机制**：`defaultProjectTrust: "ask"` 防止仓库静默更改设置

### 17.6 Pi 独特优势

1. **树形会话**：单文件存所有分支，`/tree` 原地探索替代方案
2. **消息队列 steer/follow-up**：agent 工作时可排队消息
3. **四运行模式**：交互/print/JSON/RPC/SDK，便于嵌入其他应用
4. **热重载**：`/reload` 无需重启应用扩展改动
5. **可让 pi 修改自身**：extension 可操作 pi 自身，`/reload` 后继续
6. **Skills 跨 harness 兼容**：实现 Agent Skills 标准，可复用 Claude Code/Codex skills
7. **包分发**：extensions/skills/prompts/themes 打包为 npm/git 包分享
8. **自定义 provider 强大**：extension 可实现任意 API/OAuth/自定义流式
9. **Context engineering**：最小系统提示，可通过 SYSTEM.md/APPEND_SYSTEM.md/extension 精细控制上下文

### 17.7 哲学差异

**Pi**：
- **Primitives, not features**：提供原语，而非功能
- **极简核心**：核心只做必要的事
- **强扩展性**：工作流特定行为推到 extensions/skills/packages
- **用户掌控**：「There are many agent harnesses but this one is yours」

**Claude Code/Codex/OpenCode**：
- **功能完备内置**：MCP、sub-agents、权限弹窗、plan mode 等开箱即用
- **降低门槛**：用户无需自己实现这些功能
- **官方维护**：由厂商维护功能

### 17.8 选型建议

**选 Pi 的场景**：

- 喜欢极简、可定制的工作流
- 需要树形会话探索替代方案
- 需要嵌入其他应用（SDK/RPC）
- 想用多个 LLM provider
- 需要热重载快速迭代 extension
- 想要 MIT 许可的开源方案
- 喜欢用 tmux 管理多 agent

**选 Claude Code/Codex/OpenCode 的场景**：

- 想要开箱即用的功能（MCP、sub-agents、权限弹窗）
- 不想自己实现 extension
- 已订阅对应平台（Claude Pro/Max、ChatGPT Plus/Pro）
- 喜欢内置 plan mode、to-dos 等

**Pi 与其他并用**：

- Pi 做需要定制的场景，其他做日常编码
- 利用 Pi 的 skills 跨 harness 兼容，平滑迁移
- 用 Pi 的 SDK 构建自定义工具，其他做日常开发
