# 从入门到精通

## 1 模型与 Reasoning

Codex 支持多种 GPT 模型，选择合适的模型和 reasoning effort 直接影响任务质量、速度和成本。

### 1.1 GPT-5.6 系列（推荐）

GPT-5.6 是当前旗舰系列，分三档定位：

| 模型 | 定位 | 命令 |
|------|------|------|
| **gpt-5.6-sol** | 旗舰，复杂编码、computer use、研究、网络安全 | `codex -m gpt-5.6-sol` |
| **gpt-5.6-terra** | 平衡型，性能与 GPT-5.5 相当但成本更低 | `codex -m gpt-5.6-terra` |
| **gpt-5.6-luna** | 快速低成本，清晰可重复任务 | `codex -m gpt-5.6-luna` |

选型建议：

- 日常编码用 `gpt-5.6-terra`（平衡型，性价比最高）
- 架构设计、复杂调试用 `gpt-5.6-sol`（旗舰，最强推理）
- 批量重复任务、subagent 委派用 `gpt-5.6-luna`（快速低成本）

### 1.2 其他可用模型

| 模型 | 说明 |
|------|------|
| `gpt-5.5` | 上一代前沿模型 |
| `gpt-5.4` | 专业工作前沿模型 |
| `gpt-5.4-mini` | 高效 mini 模型，适合 subagent |
| `gpt-5.3-codex-spark` | 文本-only 研究预览，近乎实时迭代（ChatGPT Pro）|

### 1.3 已弃用模型

`gpt-5.2`、`gpt-5.3-codex` 在 ChatGPT 登录下已弃用；API 中可能仍可用，需 API Key 认证。新项目不要使用。

### 1.4 Reasoning Effort 级别

Reasoning effort 控制模型思考深度，七档可选：

| 级别 | 适用场景 |
|------|---------|
| `minimal` / `none` | 模型支持时的最低延迟 |
| `low` | 简单、well-scoped 任务 |
| `medium`（默认）| 平衡速度与深度 |
| `high` | 复杂逻辑、边界条件、审慎检查 |
| `xhigh` | 极复杂推理 |
| `max` | 最难问题，深度优先 |
| `ultra` | 自动任务委派（subagent），需支持模型 |

CLI 中通过 `/reasoning` 选择，或启动时指定：

```bash
$ codex exec -m gpt-5.6 --config model_reasoning_effort='"high"' "Review the current changes"
```

### 1.5 四种设置方式

**方式一：`/model` 命令（最快捷）**

在 TUI 中输入 `/model`，弹出交互式模型选择器，选择后立即生效。

**方式二：`--model` 启动参数**

```bash
$ codex -m gpt-5.6-sol          # 本次会话使用旗舰
$ codex -m gpt-5.6-terra        # 本次会话使用平衡型
$ codex -m gpt-5.6-luna         # 本次会话使用快速型
```

**方式三：config.toml**

```toml
# ~/.codex/config.toml
model = "gpt-5.6-terra"
model_reasoning_effort = "high"
```

**方式四：环境变量**

```bash
$ export CODEX_HOME=~/.codex
# 模型通过 config.toml 或 --model 设置，无独立环境变量
```

> **注意**：Codex 不像 Claude Code 那样有 `ANTHROPIC_MODEL` 环境变量，模型通过 `config.toml` 或 `--model` 设置。Provider 相关的认证用 `env_key` 指定的环境变量（如 `OPENAI_API_KEY`）。

### 1.6 Fast Mode

Fast mode 加速输出 1.5x，但信用消费倍率更高（GPT-5.5 2.5x、GPT-5.4 2x）。仅 ChatGPT 登录可用；API Key 走标准 API 计价。

```bash
# TUI 中切换
/fast on       # 启用
/fast off      # 关闭
/fast status   # 查看状态
```

```toml
# config.toml
service_tier = "fast"

[features]
fast_mode = true
```

### 1.7 model_verbosity 与 model_reasoning_summary

控制模型输出的详细程度和推理摘要：

```toml
model_verbosity = "medium"           # low | medium | high
model_reasoning_summary = "auto"     # auto | concise | detailed | none
```

- `model_verbosity`：low 简洁、medium 平衡、high 详细
- `model_reasoning_summary`：控制是否显示推理过程摘要，`none` 完全隐藏

### 1.8 model_providers 自定义 provider 概览

Codex 支持接入任意 OpenAI Responses/Chat Completions 兼容的 provider（Ollama、LM Studio、Mistral、Azure、Bedrock 等）。详细配置见第 13 章。

```toml
# 简单示例：接入本地 Ollama
[model_providers.local_ollama]
name = "Ollama"
base_url = "http://localhost:11434/v1"

# 使用
$ codex --provider local_ollama -m qwen3-coder
```

## 2 沙箱模式

Codex 用平台原生沙箱（macOS Seatbelt、Linux bwrap+seccomp、Windows sandbox）隔离 agent 的文件和网络访问，这是与 Claude Code 最大的差异之一--Codex 有 OS 级沙箱，Claude Code 没有。

### 2.1 三种模式速查

| 模式 | 行为 | 启动参数 |
|------|------|----------|
| `read-only` | 只读文件，不能编辑或运行需审批的命令 | `--sandbox read-only` |
| `workspace-write`（默认）| 工作区内可读写、运行常规命令；网络默认关闭 | `--sandbox workspace-write` |
| `danger-full-access` | 无沙箱限制，完全访问文件系统和网络 | `--sandbox danger-full-access` |

```toml
# config.toml 默认
sandbox_mode = "workspace-write"
```

### 2.2 workspace-write 受保护路径

即使在 `workspace-write` 模式下，以下路径仍保持只读（递归保护）：

| 路径 | 说明 |
|------|------|
| `/.git` | Git 仓库元数据（目录或文件；若为 `gitdir:` 指针，解析后的路径也保护）|
| `/.agents` | agents 配置目录 |
| `/.codex` | Codex 配置目录 |

<span style="color:red;font-weight:bold;">这些保护不可绕过，防止 agent 篡改版本控制或自身配置</span>。

### 2.3 sandbox_workspace_write 配置

`workspace-write` 模式下的精细配置：

```toml
[sandbox_workspace_write]
network_access = false                    # 默认禁网
exclude_tmpdir_env_var = false            # 允许 $TMPDIR
exclude_slash_tmp = false                 # 允许 /tmp
writable_roots = ["/Users/YOU/.pyenv/shims"]  # 额外可写目录
```

常见场景：

```toml
# 场景1：允许 npm/pnpm 访问全局缓存
[sandbox_workspace_write]
writable_roots = ["~/.npm", "~/.pnpm-store"]

# 场景2：允许网络访问（如下载依赖）
[sandbox_workspace_write]
network_access = true
```

### 2.4 codex sandbox 测试命令

在 Codex 沙箱中运行任意命令，测试沙箱策略是否生效：

```bash
# macOS（Seatbelt）
$ codex sandbox macos --permissions-profile <name> --log-denied ls /etc

# Linux（Landlock）
$ codex sandbox linux --permissions-profile <name> ls /etc

# Windows
$ codex sandbox windows --permissions-profile <name> ls /etc

# 别名
$ codex sandbox seatbelt    # 同 macos
$ codex sandbox landlock    # 同 linux
$ codex debug               # 同 codex sandbox
```

`--log-denied` 打印被沙箱拒绝的操作，调试权限问题非常有用。

### 2.5 --yolo 危险模式

<span style="color:red;font-weight:bold;">完全跳过审批和沙箱，仅在已加固环境（Docker/CI/隔离虚拟机）使用</span>：

```bash
$ codex --dangerously-bypass-approvals-and-sandbox
# 别名
$ codex --yolo
```

常见合理场景：

```bash
# Docker 容器内的一次性任务
$ docker run --rm -v $(pwd):/work codex --yolo "refactor this module"

# GitHub Actions 中的 CI 任务
$ codex exec --yolo "fix the lint errors"
```

<span style="color:red;font-weight:bold;">绝对不要在本地开发环境直接用 --yolo，agent 可能误删文件或泄露密钥</span>。

## 3 审批策略

审批策略控制 Codex 何时暂停征求用户批准，与沙箱模式共同界定 agent 自主性。

### 3.1 三种策略速查

| 策略 | 行为 | 启动参数 |
|------|------|----------|
| `untrusted` | 不在信任集合中的命令都要询问 | `--ask-for-approval untrusted` |
| `on-request`（默认）| 沙箱内自主工作，越界时询问 | `--ask-for-approval on-request` |
| `never` | 不暂停审批 | `--ask-for-approval never` |

```toml
# config.toml 默认
approval_policy = "on-request"
```

### 3.2 粒度审批（Granular）

需要对不同类别的操作分别控制时，用粒度审批：

```toml
approval_policy = { granular = {
  sandbox_approval = true,        # 沙箱升级审批
  rules = true,                   # execpolicy prompt 规则
  mcp_elicitations = true,        # MCP elicitation 提示
  request_permissions = false,    # request_permissions 工具提示
  skill_approval = false          # skill-script 审批
} }
```

设为 `false` 的类别会被自动拒绝（fail closed），不会静默放行。

### 3.3 auto_review 自动审查 subagent

让 reviewer subagent 自动评估需要审批的请求，无需人工介入：

```toml
approvals_reviewer = "auto_review"
```

Auto-review 在不改变沙箱边界的前提下，对以下请求使用 reviewer subagent 评估：

- 沙箱升级
- 网络阻断
- `request_permissions` 提示
- 有副作用的 MCP 工具调用

**风险等级**：

| 风险等级 | 处理 |
|---------|------|
| 低 | 放行 |
| 中 | 放行 |
| 高 | 需足够授权且无 deny 规则 |
| critical | 拒绝 |

自动审查策略代码库开源：`codex-rs/core/src/guardian/policy.md`。

### 3.4 Smart approvals

启用时（默认），Codex 在升级请求中可能主动建议 `prefix_rule`，用户审查后接受，下次相同命令自动放行。

例如，Codex 想运行 `npm test` 但不在信任集合，会建议：

```python
prefix_rule(
  pattern = ["npm", "test"],
  decision = "allow",
  justification = "Run project tests",
)
```

用户接受后，后续 `npm test` 不再询问。

### 3.5 常见 sandbox + approval 组合

| 意图 | Flags | 效果 |
|------|------|------|
| Auto（预设）| 默认或 `--sandbox workspace-write --ask-for-approval on-request` | 工作区内自主，越界需审批 |
| 安全只读浏览 | `--sandbox read-only --ask-for-approval on-request` | 只能读、答；改/执行/网络需审批 |
| 只读非交互（CI）| `--sandbox read-only --ask-for-approval never` | 仅读，永不询问 |
| 自动编辑但 untrusted 命令需审批 | `--sandbox workspace-write --ask-for-approval untrusted` | 可读写，但非信任命令需审批 |
| Auto-review | `--sandbox workspace-write --ask-for-approval on-request -c approvals_reviewer=auto_review` | 同标准 on-request，但用自动审查 |
| 危险全访问 | `--dangerously-bypass-approvals-and-sandbox`（`--yolo`）| <span style="color:red;">无沙箱、无审批（不推荐）</span> |

### 3.6 Reviewer 配置

```toml
approvals_reviewer = "user"         # 默认：人工审查
approvals_reviewer = "auto_review"  # 自动审查 subagent
```

`user` 模式下，所有需审批的请求都暂停等待用户决定；`auto_review` 模式下，由 reviewer subagent 评估风险等级后决定。

## 4 基本交互（TUI）

### 4.1 启动与基本操作

```bash
$ codex                    # 在当前目录启动 TUI
$ codex "explain this codebase to me"   # 带初始 prompt 启动
$ codex -C /path/to/project            # 指定工作目录
$ codex --add-dir /tmp/extra           # 额外授予 /tmp/extra 写权限
```

启动后进入交互式 TUI，底部是 composer（输入框），上方是对话区和工具调用展示。

### 4.2 Steering vs Queue

Codex 运行中（turn 进行时）你仍可以发送消息，有两种模式：

| 模式 | 快捷键 | 行为 |
|------|--------|------|
| Steering | `Enter` | 加入当前 turn，立即影响正在进行的推理 |
| Queue | `Tab` | 排到下一 turn，当前 turn 完成后再处理 |

使用场景：

```bash
# Steering：发现 Codex 走错方向，立即纠正
# （Codex 正在改错文件）
别动 src/core/，改 src/utils/

# Queue：补充后续要求，不打断当前
# （Codex 正在写测试）
做完后顺便跑一遍 npm test
```

### 4.3 关键快捷键

| 操作 | 快捷键 |
|------|--------|
| Steering（加入当前 turn）| `Enter` |
| Queue（排到下一 turn）| `Tab` |
| 提交（composer）| `Enter` / `Ctrl+M` |
| 打开 transcript | `Ctrl+T` |
| 中断当前 turn | `F12`（chat 上下文）|
| 打开外部编辑器（长 prompt）| `Ctrl+G` |
| 切换 raw 滚动模式 | `Alt+R` |

`Ctrl+G` 会打开 `$VISUAL`（或 `$EDITOR`）配置的外部编辑器，适合编写长 prompt，保存退出后文本回到 composer。

### 4.4 好 Prompt 公式

清晰的 prompt 结构能显著提升输出质量：

```
[上下文] + [任务] + [约束] + [期望输出格式]
```

示例：

```
我在做一个 VitePress 知识库（见 docs/.vitepress/config.mjs），
需要新增 Codex 文档目录 docs/aillm/Codex/。
约束：仿照 docs/aillm/Claude/ 的风格，中文为主，代码标识符英文。
请先列出 00-引言.md 的章节大纲，不要直接写代码。
```

### 4.5 修正方向与 Think 关键词

如果 Codex 走偏了，不要等它做完，立即用 Steering 纠正：

```
# Codex 正在过度抽象
停，不要引入 BaseAgent 抽象层，直接在现有函数上改

# Codex 正在改错文件
这个改动应该在 src/utils.ts，不是 src/core.ts

# Codex 漏了边界情况
还要处理空数组的情况
```

**Think 关键词**：在 prompt 中加入 `think`、`think hard`、`think harder`、`ultrathink` 等关键词，会触发更深的推理模式（消耗更多 token 但输出更周全）。

### 4.6 AI 幻觉识别与应对

Codex 偶尔会：

- 编造不存在的 API 或配置字段
- 引用不存在的文件路径
- 声称已测试但实际未运行

应对策略：

1. **关键操作要求证据**：让 Codex 提供文件路径和行号，`cat -n` 验证
2. **测试驱动**：先写测试，确认失败，再写实现（见 4.7）
3. **交叉验证**：重要结论让 Codex 用 `grep` / `find` 复核
4. **保持怀疑**：Codex 说"已通过测试"时，要求贴出测试输出

### 4.7 TDD 交互流

测试驱动开发（TDD）是降低 AI 幻觉的有效方式：

```
# 1. 基于预期行为编写测试
为 parseConfig 函数写单元测试，覆盖：正常路径、空输入、无效 TOML、缺少必填字段

# 2. 确认测试失败
跑一遍 npm test，确认新测试全部失败（red）

# 3. 提交测试
git add tests/parseConfig.test.ts && git commit -m "test: add parseConfig tests"

# 4. 编写实现
现在实现 parseConfig，让所有测试通过，不要改测试
```

## 5 AGENTS.md 项目指令

`AGENTS.md` 是 Codex 的项目自定义指令文件，相当于 Claude Code 的 `CLAUDE.md`，遵循开放标准 [agents.md](https://agents.md)。

### 5.1 AGENTS.md 该放什么

```markdown
# 项目名称

## 常用命令
- `pnpm docs:dev` - 启动开发服务器
- `pnpm docs:build` - 构建生产版本

## 代码风格
- 文档使用中文
- 代码标识符保持英文
- 配置示例用 TOML

## 项目规范
- 一定不要删除文档 *.md 中的图片
- 构建产物 docs/.vitepress/dist 不提交到 git

## 重要提醒
- 包管理器必须用 pnpm
- 端口 8750 用于本地开发
```

### 5.2 发现优先级

Codex 启动时构建指令链（每次 run / TUI session 启动一次）：

1. **Global scope**（`$CODEX_HOME`，默认 `~/.codex`）
   - 若 `AGENTS.override.md` 存在则读取
   - 否则读取 `AGENTS.md`
   - 仅用第一个非空文件
2. **Project scope**：从项目根（通常 Git 根）到当前工作目录逐层向下
   - 每个目录依次检查 `AGENTS.override.md` -> `AGENTS.md` -> `project_doc_fallback_filenames` 中的 fallback
   - 每个目录最多包含一个文件
3. **Merge order**：从根向下拼接，blank line 分隔；越靠近当前目录的文件越靠后（覆盖前述）

```
~/.codex/AGENTS.md                    # 全局指令
└── /project-root/AGENTS.md           # 项目根指令
    └── /project-root/src/AGENTS.md   # 子目录指令（覆盖前述）
```

### 5.3 AGENTS.override.md 临时覆盖

`AGENTS.override.md` 优先级高于 `AGENTS.md`，用于临时全局覆盖：

```markdown
# ~/.codex/AGENTS.override.md
## 临时规则
- 今天所有改动都要先跑 npm test
- 不要提交任何代码，等我确认
```

适合场景：

- 紧急修复时临时收紧规则
- 实验性工作流，不想污染主 `AGENTS.md`

### 5.4 project_doc_fallback_filenames 自定义

如果项目用的是其他文件名（如 `TEAM_GUIDE.md`、`.agents.md`），可以配置 fallback：

```toml
# ~/.codex/config.toml
project_doc_fallback_filenames = ["TEAM_GUIDE.md", ".agents.md"]
project_doc_max_bytes = 65536   # 默认 32 KiB
```

`project_doc_max_bytes` 限制单个指令文件的最大字节，防止过大文件污染上下文。

### 5.5 /init 自动生成脚手架

在 TUI 中运行 `/init`，Codex 会扫描当前项目并生成 `AGENTS.md` 脚手架：

```
/init
```

生成的脚手架包含项目结构、常用命令、代码风格等基础信息，用户再手动完善。

### 5.6 验证当前生效的指令文件

```bash
# 让 Codex 总结当前生效的指令
$ codex --ask-for-approval never "Summarize the current instructions."

# 指定子目录查看哪些指令文件生效
$ codex --cd src --ask-for-approval never "Show which instruction files are active."
```

启用 plaintext 日志查看加载了哪些文件：

```bash
$ codex -c log_dir=./.codex-log
$ tail -F ./.codex-log/codex-tui.log
```

### 5.7 自定义 CODEX_HOME

```bash
# 用自定义的 CODEX_HOME（如多账号切换）
$ CODEX_HOME=$(pwd)/.codex codex exec "List active instruction sources"
```

### 5.8 全局指令示例

```markdown

- Prefer `pnpm` when installing dependencies.
- Ask for confirmation before adding new production dependencies.
```

## 6 config.toml 配置体系

### 6.1 ~/.codex/ 目录速览

```
~/.codex/
├── config.toml              # 用户级主配置
├── auth.json                # 凭据（若用 file 存储）
├── history.jsonl            # 会话历史（若启用）
├── AGENTS.md                # 全局自定义指令
├── AGENTS.override.md       # 临时全局覆盖
├── hooks.json               # 全局 hooks
├── rules/
│   └── default.rules        # 全局命令规则（Starlark）
├── agents/                  # 自定义子代理
│   ├── reviewer.toml
│   └── pr-explorer.toml
├── <profile-name>.config.toml  # Profile 配置层
├── memories/                # 本地 memory 文件
├── sessions/                # session rollout 文件
├── worktrees/               # Codex-managed worktrees
├── packages/standalone/     # 独立包缓存
├── log/                     # 日志（默认）
├── themes/                  # 自定义 .tmTheme
└── skills/                  # 用户级 skills（也在 ~/.agents/skills/）
```

### 6.2 配置优先级链

从高到低：

1. **CLI flags 和 `--config` 覆盖**（最高）
2. **Project 配置**：`.codex/config.toml`，从项目根到当前工作目录（最近的胜出；仅 trusted 项目）
3. **Profile 文件**：`--profile profile-name` 加载 `~/.codex/profile-name.config.toml`
4. **User 配置**：`~/.codex/config.toml`
5. **System 配置**（Unix）：`/etc/codex/config.toml`
6. **Built-in 默认值**（最低）

```
CLI flags（--model gpt-5.6）          ← 最高
    ↓
settings.local.json                  ← 无（Codex 无此概念）
    ↓
project .codex/config.toml           ← 项目级
    ↓
~/.codex/<profile>.config.toml       ← Profile 层
    ↓
~/.codex/config.toml                 ← 用户级
    ↓
/etc/codex/config.toml               ← 系统级（Unix）
    ↓
Built-in 默认                         ← 最低
```

**关键安全规则**：项目 `.codex/config.toml` 无法覆盖以下键（会被忽略并警告）：

`openai_base_url`、`chatgpt_base_url`、`apps_mcp_product_sku`、`model_provider`、`model_providers`、`notify`、`profile`、`profiles`、`experimental_realtime_ws_base_url`、`otel`

未受信任的项目会跳过所有 `.codex/` 层，包括 config、hooks、rules；user 和 system 层仍加载。

### 6.6 项目 .codex/config.toml 的安全限制

项目级配置文件 `.codex/config.toml` 虽然方便团队共享，但出于安全考虑，以下 key 会被忽略并警告：

| 被忽略的 key | 原因 |
|-------------|------|
| `openai_base_url` | 防止项目重定向 API 到恶意端点 |
| `chatgpt_base_url` | 同上 |
| `model_provider` | 防止项目强制使用非预期 provider |
| `model_providers` | 防止项目注入恶意 provider 配置 |
| `apps_mcp_product_sku` | 商业配置 |
| `notify` | 防止项目配置外部通知泄露数据 |
| `profile` / `profiles` | 防止项目强制 profile |
| `experimental_realtime_ws_base_url` | 防止项目重定向实时 WebSocket |
| `otel` | 防止项目注入恶意遥测端点 |

未受信任的项目（首次进入时未确认信任）会跳过所有 `.codex/` 层，包括 config、hooks、rules。

## 7 codex exec 非交互模式

`codex exec` 是 Codex 的非交互式入口，适合脚本化、CI/CD、批量处理。与 `claude -p` 类似但功能更丰富。

### 7.1 基本用法

```bash
# 流式输出到 stdout，进度到 stderr
$ codex exec "summarize the repository structure"

# 输出重定向到文件
$ codex exec "generate release notes for the last 10 commits" | tee release-notes.md
```

### 7.2 stdin 管道（codex exec -）

`codex exec -` 把 stdin 作为完整 prompt：

```bash
# 从文件读取 prompt
$ cat prompt.txt | codex exec -

# 组合上下文
$ printf "Summarize this error log:\n\n%s\n" "$(tail -n 200 app.log)" | codex exec -

# 管道式处理
$ curl -s https://jsonplaceholder.typicode.com/comments \
  | codex exec "format the top 20 items into a markdown table" \
  > table.md
```

### 7.3 JSONL 输出与事件类型

```bash
$ codex exec --json "summarize the repo structure" | jq
```

JSONL 事件类型：

| 事件 | 说明 |
|------|------|
| `thread.started` | 含 `thread_id` |
| `turn.started` | turn 开始 |
| `turn.completed` | turn 完成 |
| `turn.failed` | turn 失败 |
| `item.started` | item 开始（`command_execution`、`agent_message`、`file_change`、`mcp_tool_call`、`web_search`、`plan_update` 等）|
| `item.completed` | item 完成 |
| `error` | 错误 |

适合用 `jq` 过滤特定事件：

```bash
# 只看文件变更
$ codex exec --json "refactor utils" | jq 'select(.type == "item.completed" and .item.type == "file_change")'
```

### 7.4 JSON Schema 结构化输出

让 Codex 输出符合 JSON Schema 的结构化数据：

```bash
# schema 文件
$ cat schema.json
{
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "version": {"type": "string"},
    "dependencies": {"type": "array", "items": {"type": "string"}}
  },
  "required": ["name", "version"]
}

# 按 schema 输出
$ codex exec "Extract project metadata" \
  --output-schema ./schema.json \
  -o ./project-metadata.json
```

### 7.5 --ephemeral 不持久化 session

```bash
$ codex exec --ephemeral "triage this repository"
```

不写入 `~/.codex/sessions/`，适合一次性任务，避免污染 session 历史。

### 7.6 resume 继续非交互 session

```bash
# 第一次运行
$ codex exec "review the change for race conditions"

# 继续最近一次非交互 session
$ codex exec resume --last "fix the race conditions you found"

# 继续指定 session
$ codex exec resume <session-id> "..."
```

### 7.7 默认 read-only 沙箱与权限升级

<span style="color:red;font-weight:bold;">`codex exec` 默认运行在 read-only 沙箱</span>，要允许写操作需显式指定：

```bash
# 允许编辑工作区
$ codex exec --sandbox workspace-write "fix the typo in README"

# 完全访问（仅隔离环境）
$ codex exec --sandbox danger-full-access "..."
```

### 7.8 常用 flag

| Flag | 说明 |
|------|------|
| `--json` | JSONL 流式输出 |
| `--output-schema` | 按 JSON Schema 输出结构化数据 |
| `-o` / `--output-file` | 写最终消息到文件 |
| `--ephemeral` | 不持久化 session |
| `--skip-git-repo-check` | 跳过 Git 仓库检查（非 Git 目录使用）|
| `--ignore-user-config` | 跳过用户配置 |
| `--ignore-rules` | 跳过 rules |
| `--sandbox` | 沙箱策略（默认 read-only）|
| `--ask-for-approval` | 审批策略 |

### 7.9 CI/CD 集成示例

**单次运行使用不同 API Key**：

```bash
$ CODEX_API_KEY=<key> codex exec --json "triage open bug reports"
```

<span style="color:red;font-weight:bold;">不要 job-wide 设置 `CODEX_API_KEY`，inline 设置更安全</span>（避免 checkout 的代码访问到 key）。

**GitHub Actions 示例**：

```yaml
- name: Run Codex
  env:
    CODEX_API_KEY: ${{ secrets.OPENAI_API_KEY }}
  run: |
    codex exec --json "review the changes in this PR" > codex-output.json
```

推荐使用官方 `openai/codex-action`（见第 14 章 Git 工作流）。

## 8 MCP 集成

MCP（Model Context Protocol）让 Codex 连接外部工具和数据源。Codex 的 MCP 配置直接写在 `config.toml` 中，不像 Claude Code 用独立的 `.mcp.json`。

### 8.1 支持的 MCP 功能

- **STDIO 服务器**：本地进程（命令启动）
- **Streamable HTTP 服务器**：远程地址
- **环境变量**：传递给 STDIO server
- **Bearer token 认证**：HTTP server
- **OAuth 认证**：HTTP server，`codex mcp login` 触发
- **ChatGPT session 认证**：可信第一方服务器
- **Server instructions**：读取 MCP `instructions` 字段作为 server-wide guidance（前 512 字符尤其重要）

### 8.2 codex mcp 命令

```bash
# 添加 STDIO server
$ codex mcp add <name> -- <command> [args...]

# 示例：Context7
$ codex mcp add context7 -- npx -y @upstash/context7-mcp

# 带环境变量
$ codex mcp add --env VAR1=VALUE1 --env VAR2=VALUE2 -- npx -y some-mcp-server

# 列出已配置服务器
$ codex mcp list

# OAuth 登录
$ codex mcp login <name>

# 查看所有命令
$ codex mcp --help
```

### 8.3 config.toml 配置 MCP server

#### STDIO server

```toml
[mcp_servers.context7]
command = "npx"
args = ["-y", "@upstash/context7-mcp"]
env_vars = ["LOCAL_TOKEN"]
env = { MY_ENV_VAR = "MY_ENV_VALUE" }
cwd = "/path/to/working/dir"
startup_timeout_sec = 10
tool_timeout_sec = 60
enabled = true
required = false                   # true 时启动失败会终止 Codex
enabled_tools = ["tool1", "tool2"] # 允许列表
disabled_tools = ["tool3"]         # 禁用列表（在 enabled_tools 后应用）
default_tools_approval_mode = "prompt"  # auto|prompt|writes|approve
```

#### Streamable HTTP server

```toml
[mcp_servers.figma]
url = "https://mcp.figma.com/mcp"
auth = "oauth"                     # oauth（默认）|chatgpt
bearer_token_env_var = "FIGMA_OAUTH_TOKEN"
http_headers = { "X-Figma-Region" = "us-east-1" }
env_http_headers = { "X-Header" = "ENV_VAR_NAME" }
```

#### OAuth 回调配置

```toml
mcp_oauth_callback_port = 5555
mcp_oauth_callback_url = "https://devbox.example.internal/callback"
mcp_oauth_credentials_store = "auto"   # auto|file|keyring
```

### 8.4 每工具审批覆盖

对单个工具的审批行为进行精细控制：

```toml
[mcp_servers.chrome_devtools.tools.open]
approval_mode = "approve"          # auto|prompt|writes|approve
```

### 8.5 在会话中查看

TUI 中输入 `/mcp` 查看已连接服务器和工具状态。桌面应用中同样在 `/mcp` 面板查看。

### 8.6 Codex 作为 MCP server

Codex 本身可以作为 MCP server 暴露给其他客户端：

```bash
$ codex mcp-server
```

暴露两个工具：

| 工具 | 说明 |
|------|------|
| `codex` | 启动新会话（prompt + 配置覆盖：approval-policy、base-instructions、compact-prompt、config、cwd、developer-instructions、model、sandbox）|
| `codex-reply` | 通过 `threadId` 继续会话 |

用 MCP Inspector 测试：

```bash
$ npx @modelcontextprotocol/inspector codex mcp-server
```

可作为 OpenAI Agents SDK 的 MCP 后端构建多 agent 工作流。

### 8.7 Plugin 提供的 MCP server

## 9 Subagent

Subagent 让 Codex 委派独立任务给子 agent，避免主会话上下文污染。与 Claude Code 的 Task 工具类似，但 Codex 的 subagent 系统更完整。

### 9.1 内置 Agent

| 名称 | 说明 |
|------|------|
| `default` | 通用 fallback agent |
| `worker` | 执行导向，实现和修复 |
| `explorer` | 只读代码库探索 |

### 9.2 自定义 Agent 文件

位置：
- 用户级：`~/.codex/agents/<name>.toml`
- 项目级：`.codex/agents/<name>.toml`

**必填字段**：
- `name`：agent 名称（Codex 用此识别，文件名匹配只是约定）
- `description`：人类可读，何时使用此 agent
- `developer_instructions`：核心行为指令

**可选字段**（省略时继承父 session）：
- `nickname_candidates`：显示昵称池
- `model`、`model_reasoning_effort`、`sandbox_mode`
- `mcp_servers`、`skills.config`

### 9.3 自定义 Agent 示例

**PR review 三件套**：

```toml
# .codex/config.toml
```

### 9.4 全局 [agents] 配置

```toml
[agents]
max_threads = 6                  # 并发线程上限（默认 6）
max_depth = 1                    # 嵌套深度（root=0；默认 1 防递归）
job_max_runtime_seconds = 1800   # spawn_agents_on_csv 默认超时
interrupt_message = true         # turn 中断时记录模型可见消息
```

### 9.5 Subagent 工作流

**关键概念**：
- Subagent workflow：并行运行多个 agent 并合并结果
- Subagent：被委派的 agent
- Agent thread：subagent 工作的线程（可打开查看进度）

**为什么有用**：
- 避免 context pollution（中间噪声压住关键信息）
- 避免 context rot（长对话性能下降）
- 并行独立工作，节省时间

**触发方式**：
- 多数 intelligence level 需显式要求："spawn two agents"、"delegate this work in parallel"
- Ultra mode（仅 ChatGPT Work）可主动委派

### 9.6 触发示例

```
Review this branch with parallel subagents. Spawn one subagent for security risks,
one for test gaps, and one for maintainability. Wait for all three, then summarize
the findings by category with file references.
```

或者调用自定义 agent：

```
Review this branch against main. Have pr_explorer map the affected code paths,
reviewer find real risks, and docs_researcher verify the framework APIs that
the patch relies on.
```

### 9.7 审批与沙箱继承

- Subagent 继承父 turn 的 sandbox policy 与 approval mode
- 父 turn 的实时覆盖（如 `/permissions` 改动、`--yolo`）也会传给子 agent
- 自定义 agent 文件中的 `sandbox_mode` 可单独覆盖

### 9.8 CSV 批处理（实验性）

`spawn_agents_on_csv` 工具：每行一个工作项，并行处理。

```
Create /tmp/components.csv with columns path,owner and one row per frontend component.
Then call spawn_agents_on_csv with:
- csv_path: /tmp/components.csv
- id_column: path
- instruction: "Review {path} owned by {owner}. Return JSON with keys path, risk, summary, follow_up via report_agent_job_result."
- output_csv_path: /tmp/components-review.csv
- output_schema: an object with required string fields path, risk, summary, follow_up
```

每个 worker 必须调用 `report_agent_job_result` 一次；导出 CSV 含原始行 + `job_id`、`item_id`、`status`、`last_error`、`result_json`。

### 9.9 管理 subagent

- TUI：`/agent` 切换 active agent 线程
- 桌面应用：Subagents 面板，Active / Done 列表
- 直接告诉 Codex 调整、停止或关闭 subagent 线程

## 10 Hooks

Hooks 是 Codex 生命周期中的确定性脚本扩展点，可在关键节点插入自定义逻辑。

### 10.1 Hook 事件列表

| 事件 | 触发时机 | matcher 过滤 |
|------|---------|-------------|
| `SessionStart` | session 启动 | `source`：`startup` / `resume` / `clear` / `compact` |
| `SubagentStart` | subagent 启动 | `agent_type` |
| `PreToolUse` | 工具调用前 | tool name |
| `PermissionRequest` | 即将请求审批时 | tool name |
| `PostToolUse` | 工具调用后 | tool name |
| `UserPromptSubmit` | 用户提交 prompt | 不支持 matcher |
| `PreCompact` | 压缩前 | compaction trigger：`manual` / `auto` |
| `PostCompact` | 压缩后 | compaction trigger |
| `SubagentStop` | subagent 停止 | subagent type |
| `Stop` | turn 停止 | 不支持 matcher |

`matcher` 是正则字符串，`"*"` / `""` / 省略 = 匹配全部。

### 10.2 配置位置

Codex 在以下位置发现 hooks：
- `~/.codex/hooks.json`
- `~/.codex/config.toml`（内联 `[hooks]`）
- `<project>/.codex/hooks.json`（仅 trusted 项目）
- `<project>/.codex/config.toml`（内联）
- 插件 manifest 或 `hooks/hooks.json`

多来源 hooks 全部运行（不替换）；同事件多 hook 并发启动。

### 10.3 配置 schema

**JSON 格式**（`~/.codex/hooks.json`）：

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "python3 ~/.codex/hooks/session_start.py",
            "statusMessage": "Loading session notes"
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/pre_tool_use_policy.py\"",
            "statusMessage": "Checking Bash command"
          }
        ]
      }
    ],
    "PermissionRequest": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/permission_request.py\""
        }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/post_tool_use_review.py\""
        }]
      }
    ],
    "UserPromptSubmit": [
      { "hooks": [{
        "type": "command",
        "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/user_prompt_submit.py\""
      }]}
    ],
    "Stop": [
      { "hooks": [{
        "type": "command",
        "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/stop_continue.py\"",
        "timeout": 30
      }]}
    ]
  }
}
```

**字段说明**：

| 字段 | 说明 |
|------|------|
| `type` | 目前仅 `"command"` 支持（`prompt` / `agent` 解析但跳过）|
| `command` | shell 命令 |
| `commandWindows` / `command_windows` | Windows 专用覆盖 |
| `timeout` | 秒，默认 600 |
| `statusMessage` | 可选状态消息 |
| `async` | 解析但不支持 |

工作目录为 session `cwd`。

**TOML 等价**（`config.toml` 内联）：

```toml
[[hooks.PreToolUse]]
matcher = "^Bash$"

[[hooks.PreToolUse.hooks]]
type = "command"
command = '/usr/bin/python3 "$(git rev-parse --show-toplevel)/.codex/hooks/pre_tool_use_policy.py"'
timeout = 30
statusMessage = "Checking Bash command"
```

### 10.4 Hook 输入（stdin JSON）

通用字段：

| 字段 | 说明 |
|------|------|
| `session_id` | Codex session id（subagent hooks 用父 session id）|
| `transcript_path` | transcript 文件路径（可能为 null）|
| `cwd` | session 工作目录 |
| `hook_event_name` | 当前事件名 |
| `model` | Codex 扩展字段，活动模型 slug |
| `turn_id` | turn-scoped hooks 的 Codex 扩展字段 |
| `permission_mode` | `default` / `acceptEdits` / `plan` / `dontAsk` / `bypassPermissions` |

事件特定字段：

| 事件 | 特定字段 |
|------|---------|
| `SessionStart` | `source`：`startup` / `resume` / `clear` / `compact` |
| `SubagentStart` | `agent_id`、`agent_type` |
| `PreToolUse` | `tool_name`、`tool_use_id`、`tool_input`（`Bash` 和 `apply_patch` 用 `tool_input.command`）|
| `PermissionRequest` | `tool_name`、`tool_input`、`tool_input.description` |

### 10.5 Hook 输出（stdout JSON）

通用字段：

| 字段 | 说明 |
|------|------|
| `continue` | `false` 标记停止 |
| `stopReason` | 停止原因 |
| `systemMessage` | UI 警告或事件流 |
| `suppressOutput` | 解析但未实现 |

**PreToolUse 特殊输出**：

拒绝工具调用：

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Destructive command blocked by hook."
  }
}
```

或老式：

```json
{ "decision": "block", "reason": "..." }
```

或 exit code 2 + stderr 写理由。

添加模型可见上下文不阻断：

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "additionalContext": "The pending command touches generated files."
  }
}
```

重写工具调用：

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow",
    "updatedInput": { "command": "echo rewritten" }
  }
}
```

（Bash/apply_patch 需含 `command` 字段；MCP 工具为参数对象；仅 `allow` 时支持）

**PermissionRequest 输出**：可 allow / deny / 不决定（让正常审批继续）。

**SessionStart / SubagentStart**：`hookSpecificOutput.additionalContext` 加入 developer context。

**PostToolUse**：支持 `systemMessage`、`continue: false`、`stopReason`。

### 10.6 Hook 信任审查

非托管 hooks 必须审查并信任后才能运行。Codex 按 hook 当前 hash 记录信任：

- TUI：`/hooks` 查看来源、审查新/变更 hooks、信任、禁用
- 启动时若需审查会打印警告
- 变更后的 hook 重新标记需审查
- 一次性自动化可 `--dangerously-bypass-hook-trust`

**Managed hooks**（来自 system/MDM/cloud/`requirements.toml`）：标记为 managed，按策略信任，不可从用户 hook 浏览器禁用。

### 10.7 插件捆绑 hooks

```json
// .codex-plugin/plugin.json
{
  "name": "repo-policy",
  "hooks": "./hooks/hooks.json"
}
```

manifest `hooks` 可为路径、路径数组、内联对象、内联对象数组。

环境变量：
- `PLUGIN_ROOT`：插件根
- `PLUGIN_DATA`：插件可写数据目录
- 兼容 `CLAUDE_PLUGIN_ROOT`、`CLAUDE_PLUGIN_DATA`

### 10.8 关闭 hooks

```toml
[features]
hooks = false      # codex_hooks 是已弃用别名
```

### 10.9 实战示例

#### 示例1：API key 泄露扫描

```python
# ~/.codex/hooks/pre_tool_use_policy.py
import sys, json, re

data = json.load(sys.stdin)
command = data.get("tool_input", {}).get("command", "")

# 检测疑似 API key
if re.search(r'(sk-[a-zA-Z0-9]{20,}|AKIA[A-Z0-9]{16})', command):
    print(json.dumps({
        "hookSpecificOutput": {
            "hookEventName": "PreToolUse",
            "permissionDecision": "deny",
            "permissionDecisionReason": "Possible API key in command."
        }
    }))
    sys.exit(0)
```

#### 示例2：对话日志发送到外部系统

```python
# ~/.codex/hooks/stop_continue.py

data = json.load(sys.stdin)
requests.post("https://logs.example.com/codex", json={
    "session_id": data["session_id"],
    "cwd": data["cwd"],
    "timestamp": data.get("turn_id")
})
```

#### 示例3：按目录定制 prompt

```python

cwd = data["cwd"]

# 在 frontend 目录下加载额外指令

    print(json.dumps({
        "hookSpecificOutput": {

        }
    }))
```

## 11 Skills

Skills 是 Codex 的可复用指令包，遵循开放 [agent skills standard](https://agentskills.io)。与 Claude Code 的 Skills 系统类似，但 Codex 的 Skills 是开放标准，可跨 agent 工具复用。

### 11.1 Skill 目录结构

```
my-skill/
├── SKILL.md          # 必需：指令 + 元数据
├── scripts/          # 可选：可执行代码
├── references/       # 可选：文档
├── assets/           # 可选：模板、资源
└── agents/
    └── openai.yaml   # 可选：外观和依赖
```

### 11.2 SKILL.md frontmatter

`SKILL.md` 必须含 `name` 和 `description`：

```markdown
---
name: skill-name
description: Explain exactly when this skill should and should not trigger.
---
Skill instructions for Codex to follow.
```

`description` 字段尤其重要，Codex 用它判断何时隐式触发该 skill。

### 11.3 Progressive disclosure

Codex 启动时**只载入每个 skill 的 name + description + 路径**（最多占 2% 上下文窗口或 8000 字符），决定使用时才读完整 `SKILL.md`。

这种渐进式披露机制避免所有 skill 全量加载污染上下文。

### 11.4 Skill 位置与优先级

| 范围 | 位置 | 用途 |
|------|------|------|
| REPO | `$CWD/.agents/skills` | 当前工作目录 |
| REPO | `$CWD/../.agents/skills` | 父目录（仓库内嵌套）|
| REPO | `$REPO_ROOT/.agents/skills` | 仓库根 |
| USER | `$HOME/.agents/skills` | 用户全局 |
| ADMIN | `/etc/codex/skills` | 系统范围 |
| SYSTEM | 内置 | OpenAI 提供（skill-creator、plan 等）|

### 11.5 显式调用与隐式调用

**显式调用**：在 composer 中输入 `$skill-name`：

```
$skill-creator 帮我创建一个 code-review skill
```

**隐式调用**：Codex 根据 `description` 匹配自动选择。写好 `description` 是隐式触发的关键。

### 11.6 $skill-creator 内置创建器

Codex 内置 `skill-creator` skill，交互式引导创建新 skill：

```
$skill-creator 我想创建一个自动生成 Git commit message 的 skill
```

`skill-creator` 会询问 skill 名称、触发场景、指令内容，自动生成目录结构和 `SKILL.md`。

### 11.7 $skill-installer 安装 curated skills

Codex 官方维护了一批 curated skills，用 `skill-installer` 一键安装：

```bash
$ $skill-installer linear   # 安装 $linear skill
$ $skill-installer github   # 安装 $github skill
```

### 11.8 Record & Replay 录制工作流

macOS 上展示一次工作流，Codex 录制并生成可复用 skill。需要 Computer Use。初期不可用于 EEA、英国、瑞士。

适合场景：

- 重复性的 GUI 操作（如 Linear 创建 issue）
- 复杂的多步骤工作流（录制一次，以后自动执行）

### 11.9 openai.yaml 元数据（可选）

```yaml
interface:
  display_name: "Optional user-facing name"
  short_description: "..."
  icon_small: "./assets/small-logo.svg"
  icon_large: "./assets/large-logo.png"
```

### 11.10 启用/禁用

```toml

```

禁用后的 skill 不参与隐式调用，但仍可显式 `$skill-name` 调用。

## 13 第三方 Provider 接入

Codex 支持接入任意 OpenAI Responses/Chat Completions 兼容的 provider，包括本地 OSS 模型、云厂商、LLM 代理等。这是 Codex 相比 Claude Code 的优势之一--Claude Code 只能接入 Anthropic 模型或通过第三方代理。

### 13.2 --oss 模式（本地 OSS provider）

### 13.3 自定义 Provider 配置

通用字段：

| 字段 | 说明 |
|------|------|
| `name` | Provider 显示名称 |

| `env_key` | 读取的环境变量名（如 `MISTRAL_API_KEY`）|

认证两种方式（按优先级）：

1. **Bedrock API Key**：`AWS_BEARER_TOKEN_BEDROCK` + `AWS_REGION`
2. **AWS SDK 凭证链**：`aws configure` / `AWS_ACCESS_KEY_ID` 等

支持的模型 ID：
- `openai.gpt-5.5`
- `openai.gpt-5.4`

### 13.9 LLM Proxy

用 OpenAI auth 走自建代理：

```toml
# 方法1：覆盖内置 openai provider 的 base URL
openai_base_url = "https://us.api.openai.com/v1"

# 方法2：自定义 provider（用 OpenAI auth）
[model_providers.proxy]
name = "OpenAI using LLM proxy"
base_url = "http://proxy.example.com"
requires_openai_auth = true    # 用 ChatGPT 或 API Key 登录，忽略 env_key
```

### 13.10 命令式 bearer token

适合需要动态获取 token 的场景（如公司内部 token 交换服务）：

```toml
[model_providers.proxy]
name = "OpenAI using LLM proxy"
base_url = "https://proxy.example.com/v1"

[model_providers.proxy.auth]
command = "/usr/local/bin/fetch-codex-token"
args = ["--audience", "codex"]
timeout_ms = 5000
refresh_interval_ms = 300000   # 0 = 仅在认证重试时刷新
```

命令无 stdin，打印 token 到 stdout；不可与 `env_key` / `experimental_bearer_token` / `requires_openai_auth` 共用。

### 13.11 OpenAI Data Residency

数据驻留配置（如欧盟数据不出欧盟）：

```toml
model_provider = "openaidr"

[model_providers.openaidr]
name = "OpenAI Data Residency"

```

### 13.12 无认证 Provider

本地无认证的兼容 API：


## 15 上下文管理

### 15.1 /compact 压缩上下文

```bash
# TUI 中
/compact
```

智能压缩当前 task 上下文，保留关键信息，释放 50-70% token。与 Claude Code 的 `/compact` 类似。

### 15.2 /status 查看上下文用量

```bash
# TUI 中
/status
```

显示：

- task ID
- context 用量（已用 token / 总量）
- rate limit（剩余额度、重置时间）

### 15.3 自动压缩阈值

```toml
# config.toml
model_auto_compact_token_limit = 150000   # 默认 150000
model_context_window = 200000             # 强制上下文窗口
```

当上下文接近 `model_auto_compact_token_limit` 时，Codex 自动触发压缩。

### 15.4 判断压缩时机

何时该手动 `/compact`：

| 信号 | 说明 |
|------|------|
| 上下文用量 > 70% | 接近上限，建议压缩 |
| Codex 开始"忘事" | 早期约定被忽略 |
| 响应变慢 | 上下文过大导致推理延迟 |
| 切换任务方向 | 之前的话题不再相关 |

何时该 `/clear`（清除全部上下文）：

- 完全切换到不相关的任务
- 上下文严重污染，压缩也无法挽救
- 想从干净状态开始

### 15.5 /compact vs /clear

| 操作 | 效果 | 适用场景 |
|------|------|----------|
| `/compact` | 智能压缩，保留关键信息 | 上下文变大但任务延续 |
| `/clear` | 清除全部，完全重置 | 任务完全切换、上下文污染严重 |

### 15.6 好习惯

```bash
# 1. 定期检查上下文
/status

# 2. 接近上限时压缩
/compact

# 3. 完成一个大任务后清理
/clear

# 4. 长任务拆分成 subagent，避免主会话上下文膨胀
# （见第 9 章 Subagent）
```

### 15.8 Memories

Memories 是 Codex 的持久化记忆系统（默认关闭）：

```toml
[features]
memories = true                             # 必须先启用

memories.generate_memories = true           # 新 task 作为 memory 输入
memories.use_memories = true                # 注入已有 memories
memories.disable_on_external_context = false  # MCP/web search task 不参与 memory 生成
memories.min_rate_limit_remaining_percent = 25  # rate-limit 阈值
memories.extract_model = "gpt-5.4-mini"     # 单 task memory 提取模型
memories.consolidation_model = "gpt-5.4"    # 全局 memory 整合模型
```

- 启用后 Codex 从 eligible 历史 task 生成 memory 文件
- 存储在 `~/.codex/memories/`
- 自动 redact secrets
- `/memories` 控制单 task 是否使用/生成 memories

### 15.9 Chronicle（桌面应用研究预览）

Chronicle 从屏幕恢复最近工作上下文，构建 memory。ChatGPT Pro 用户 macOS 上 opt-in。

适合场景：

- 中断后恢复工作（如开会、吃饭回来）
- 跨 session 延续思路
- 自动总结工作日志

### 15.10 避免上下文污染的 subagent 分流策略

主会话上下文容易被中间噪声污染，用 subagent 分流：

```
# 让 subagent 做探索，主会话只收结论

只返回文件路径和行号列表

# 让 subagent 做实验，主会话只收结果
```

### 17.5 迁移指南

#### CLAUDE.md -> AGENTS.md

```bash
# 简单重命名
$ mv CLAUDE.md AGENTS.md
```

内容基本兼容，需调整的细节：

- 把 `Claude Code` 替换为 `Codex`
- 把 `claude` 命令替换为 `codex`
- Slash 命令大部分兼容（`/compact`、`/clear`、`/model` 等），少数不同（Claude 的 `/rewind` 在 Codex 无对应）

#### settings.json -> config.toml

格式从 JSON 转 TOML，字段映射：

```json
// settings.json (Claude)
{
  "model": "sonnet",
  "env": {
    "ANTHROPIC_BASE_URL": "http://localhost:11434"
  }
}
```

```toml
# config.toml (Codex)
model = "gpt-5.6-terra"

[model_providers.local_ollama]
name = "Ollama"
base_url = "http://localhost:11434/v1"
```

#### .claude/agents/ -> .codex/agents/

格式从 JSON 转 TOML：

```json
// .claude/agents/reviewer.json (Claude)
{
  "name": "reviewer",
  "description": "PR reviewer",
  "tools": ["Read", "Grep"]
}
```

```toml
# .codex/agents/reviewer.toml (Codex)
name = "reviewer"
description = "PR reviewer"
model = "gpt-5.4"
sandbox_mode = "read-only"
developer_instructions = """
Review code like an owner.
"""
```

#### .claude/hooks.json -> .codex/hooks.json

格式基本兼容，但事件名和字段有差异：

| Claude Code | Codex CLI |
|-------------|-----------|
| `PreToolUse` | `PreToolUse`（兼容）|
| `PostToolUse` | `PostToolUse`（兼容）|
| `UserPromptSubmit` | `UserPromptSubmit`（兼容）|
| `Stop` | `Stop`（兼容）|
| `SessionStart` | `SessionStart`（兼容）|
| - | `SubagentStart` / `SubagentStop`（Codex 独有）|
| - | `PermissionRequest`（Codex 独有）|
| - | `PreCompact` / `PostCompact`（Codex 独有）|

输出格式：Claude 用 `decision` / `reason`，Codex 用 `hookSpecificOutput.permissionDecision` / `permissionDecisionReason`（向后兼容老格式）。

#### .claude/skills/ -> .agents/skills/

路径调整：

```bash
$ mv ~/.claude/skills/my-skill ~/.agents/skills/my-skill
```

`SKILL.md` 格式基本兼容，Codex 额外支持 `agents/openai.yaml` 元数据。

### 17.6 哲学差异

**Codex CLI**：
- 强调**安全沙箱 + 平台原生隔离**，default `workspace-write` + `on-request`
- 企业级管理（`requirements.toml`、强制 login 方式、permission profiles）
- 多 surface（CLI / 桌面 / Web / IDE / Cloud）共享 config
- 开放生态（agents.md 标准、agent skills 标准、任意 provider）

**Claude Code**：
- 更轻量，依赖开发者自律 + permissions 配置
- 强调 context management（CLAUDE.md、subagents、skills）
- 单一 Anthropic 模型生态

### 17.7 选型建议

**选 Codex 的场景**：

- 需要强沙箱隔离（如运行不信任的代码、敏感项目）
- 企业团队需要统一管理和审计
- 想用 GPT-5.6 系列模型
- 需要多 provider 接入（本地 OSS + 云端混合）
- 需要 Cloud 并行任务
- 需要 Computer Use 或内置浏览器
- CI/CD 自动化（`codex exec` 功能更强）

**选 Claude Code 的场景**：

- 已订阅 Claude Pro/Max，想用满额度
- 偏好 Node.js 生态（自定义 hook、插件用 JS 写）
- 主要用 Claude 模型
- 喜欢 `CLAUDE.md` + `settings.json` 的简单配置
- 不需要复杂沙箱

**两者并用**：

- 不同项目用不同工具（按项目 `AGENTS.md` / `CLAUDE.md` 判断）
- 同一项目交替使用（如 Codex 做安全审查、Claude Code 做编码）
- 用 `cc-switch` 类工具快速切换
